# 6.4 스위치 근거리 네트워크 핵심 정리

## 스위치 근거리 네트워크

<p align="center">
  <img width="600" alt="스위치 근거리 네트워크" src="https://user-images.githubusercontent.com/76640167/214864184-93f04822-630e-4fb7-82e4-1453028bf915.png">
</p>

> 스위치는 링크 계층에서 동작하며, 링크 계층 프레임을 교환한다.

스위치는 IP 주소를 보고 경로를 결정하지 않는다.  
또한 OSPF 같은 네트워크 계층의 라우팅 알고리즘을 사용하지 않는다.

즉, 스위치는 네트워크 계층 주소가 아니라 **링크 계층 주소인 MAC 주소**를 사용해 프레임을 전달한다.

```text
스위치 -> MAC 주소 사용, 2계층 장비
라우터 -> IP 주소 사용, 3계층 장비
```

---

## 6.4.1 링크 계층 주소체계와 ARP

### 링크 계층 주소가 필요한 이유

IP 주소가 존재하는데도 MAC 주소가 필요한 이유는 계층마다 담당하는 역할이 다르기 때문이다.

- IP 주소는 네트워크 계층에서 목적지를 식별하는 주소이다.
- MAC 주소는 같은 링크 안에서 프레임을 전달하기 위한 주소이다.
- LAN은 IP만을 위해 만들어진 것이 아니라 여러 네트워크 계층 프로토콜을 지원할 수 있어야 한다.
- 계층이 독립적으로 동작하려면 각 계층은 자신만의 주소 체계를 가져야 한다.

즉, IP 주소는 네트워크 전체 관점의 주소이고, MAC 주소는 한 링크 안에서 실제 프레임 전달에 사용되는 주소이다.

---

## MAC 주소

<p align="center">
  <img width="400" alt="MAC 주소" src="https://user-images.githubusercontent.com/76640167/214864181-000d5599-c0e3-451f-98ad-266d5e54debd.png">
</p>

> MAC 주소는 링크 계층에서 사용하는 주소이며, 호스트 자체가 아니라 네트워크 인터페이스에 할당된다.

하나의 장치가 여러 개의 네트워크 인터페이스를 가지고 있다면, 그 장치는 여러 개의 MAC 주소를 가질 수 있다.  
반면 링크 계층 스위치는 프레임을 전달하는 장비이므로, 일반적으로 호스트나 라우터를 연결하는 인터페이스마다 MAC 주소를 필요로 하지는 않는다.

### 핵심 특징

- 6바이트, 48비트 주소이다.
- 보통 16진수로 표현한다.
- 네트워크 인터페이스, 즉 어댑터에 할당된다.
- 평면 구조의 주소이다.
- 위치가 바뀌어도 원칙적으로 변경되지 않는다.
- IEEE가 MAC 주소 공간을 관리한다.

```text
IP 주소  -> 위치 정보에 가까움
MAC 주소 -> 장치 식별 정보에 가까움
```

### MAC 주소 표기 예

```text
AA-BB-CC-DD-EE-FF
```

### MAC 브로드캐스트 주소

LAN 안의 모든 어댑터에게 프레임을 전달하고 싶을 때는 MAC 브로드캐스트 주소를 사용한다.

```text
FF-FF-FF-FF-FF-FF
```

## ARP(Address Resolution Protocol)

<p align="center">
  <img width="400" alt="ARP" src="https://user-images.githubusercontent.com/76640167/214864178-a1d7eb1b-8a60-47a3-8784-dc2fd974ed64.png">
</p>

> ARP는 IP 주소에 대응되는 MAC 주소를 찾아주는 프로토콜이다.

송신 호스트가 같은 서브넷에 있는 목적지로 데이터그램을 보내려면 목적지 IP 주소뿐만 아니라 목적지 MAC 주소도 알아야 한다.  
이때 ARP가 같은 서브넷 안에서 IP 주소를 MAC 주소로 변환해준다.

### ARP의 특징

- 같은 서브넷 안에서 동작한다.
- IP 주소를 MAC 주소로 변환한다.
- 인터페이스마다 ARP 모듈이 존재한다.
- 네트워크 계층과 링크 계층의 경계에 있는 프로토콜이다.

---

## ARP 테이블

> ARP 테이블은 IP 주소와 MAC 주소의 매핑 정보를 저장한다.

각 호스트와 라우터는 ARP 테이블을 가지고 있다.  
ARP 테이블에는 다음 정보가 저장된다.

- IP 주소
- MAC 주소
- TTL

ARP 테이블의 엔트리는 영구적으로 유지되지 않는다.  
일정 시간이 지나면 삭제되고, 필요할 때 다시 ARP를 통해 학습된다.

### 핵심 특징

- 자동으로 생성된다.
- TTL 기반으로 오래된 엔트리가 삭제된다.
- 관리자가 직접 구성하지 않아도 된다.
- Plug-and-Play 방식으로 동작한다.

---

## ARP 동작 과정

1. 송신 호스트가 목적지 IP 주소에 대응되는 MAC 주소를 필요로 한다.
2. ARP 테이블에 해당 정보가 없으면 ARP Request를 브로드캐스트한다.
3. 같은 서브넷의 모든 호스트가 ARP Request를 수신한다.
4. 요청된 IP 주소를 가진 호스트만 ARP Reply를 보낸다.
5. 송신 호스트는 응답을 통해 목적지 MAC 주소를 알게 된다.
6. 송신 호스트는 자신의 ARP 테이블을 갱신한다.

```text
ARP Request -> 브로드캐스트
ARP Reply   -> 유니캐스트
```

## 다른 서브넷으로 데이터그램 전송

<p align="center">
  <img width="750" alt="서브넷에 없는 노드로의 데이터그램 전송" src="https://user-images.githubusercontent.com/76640167/214864175-d885eaa2-fff3-4b6b-8d02-1ccb4e6a109a.png">
</p>

> 목적지가 다른 서브넷에 있으면 최종 목적지의 MAC 주소가 아니라 라우터 인터페이스의 MAC 주소를 사용한다.

송신 호스트는 다른 서브넷의 목적지 호스트로 직접 프레임을 보낼 수 없다.  
따라서 먼저 자신의 서브넷에 있는 라우터 인터페이스, 즉 기본 게이트웨이의 MAC 주소를 ARP로 알아낸다.

### 전송 과정

1. 송신 호스트는 라우터 인터페이스의 MAC 주소를 ARP로 알아낸다.
2. IP 데이터그램의 목적지 IP 주소는 최종 목적지로 유지한다.
3. 링크 계층 프레임의 목적지 MAC 주소는 라우터 인터페이스의 MAC 주소로 설정한다.
4. 라우터는 프레임을 받아 IP 데이터그램을 추출한다.
5. 라우터는 포워딩 테이블을 보고 다음 네트워크로 데이터그램을 전달한다.
6. 다음 링크에서 다시 ARP를 사용해 목적지 MAC 주소를 알아낸다.

## 6.4.2 이더넷

<p align="center">
  <img width="700" alt="이더넷 기술" src="https://user-images.githubusercontent.com/76640167/214864165-8bc1b17f-997a-4d55-8f1d-4b0280098251.png">
</p>

> 이더넷은 오늘날 가장 널리 사용되는 LAN 기술이다.

이더넷은 시간이 지나면서 물리적 구조가 변화했다.  
초기에는 공유 매체를 사용하는 구조였지만, 현재는 대부분 스위치 기반 구조를 사용한다.

### 발전 과정

#### 1980년대

- 동축 케이블 기반 버스 토폴로지를 사용했다.
- 브로드캐스트 방식으로 동작했다.
- 모든 프레임이 모든 어댑터를 지나갔다.
- 여러 노드가 동시에 전송하면 충돌이 발생할 수 있었다.

#### 1990년대

- 허브 기반 스타 토폴로지가 사용되었다.
- 호스트들은 꼬임쌍선을 통해 허브에 연결되었다.
- 허브는 물리 계층 장비로, 들어온 비트를 모든 포트로 복사해 전달했다.

#### 2000년대 이후

- 허브가 스위치로 대체되었다.
- 스위치 기반 스타 토폴로지가 일반화되었다.
- 스위치는 저장 후 전달 방식으로 프레임을 처리한다.
- 충돌이 거의 발생하지 않는다.

---

## 이더넷 프레임 구조

<p align="center">
  <img width="800" alt="이더넷 프레임" src="https://user-images.githubusercontent.com/76640167/214864172-56d0a3f3-1ea3-41a1-b1ae-1033039847eb.png">
</p>

> 이더넷 프레임은 링크 계층에서 데이터를 전달하기 위한 기본 단위이다.

### 주요 필드

#### 데이터 필드

- IP 데이터그램 같은 상위 계층 데이터를 담는다.
- 크기는 46~1500바이트이다.
- 1500바이트보다 크면 단편화가 필요하다.
- 46바이트보다 작으면 패딩을 추가한다.

#### 목적지 주소

- 목적지 MAC 주소가 들어간다.
- 크기는 6바이트이다.

#### 출발지 주소

- 출발지 MAC 주소가 들어간다.
- 크기는 6바이트이다.

#### 타입 필드

- 상위 계층 프로토콜을 구분한다.
- IP, ARP 등 어떤 프로토콜의 데이터인지 알려준다.

#### CRC

- 프레임에 오류가 있는지 검사한다.
- 오류가 있으면 수신 어댑터는 프레임을 폐기한다.

#### 프리앰블

- 수신 어댑터를 깨우고 송신자와 수신자의 클록을 동기화한다.
- 실제 프레임 내용이 곧 시작된다는 것을 알려준다.

---

## 이더넷의 서비스 특징

### 비연결형 서비스

이더넷은 프레임을 보내기 전에 송신자와 수신자가 연결 설정을 하지 않는다.  
즉, 핸드셰이크 없이 바로 프레임을 전송한다.

### 비신뢰적 서비스

수신 어댑터는 CRC로 오류를 검사한다.  
하지만 오류가 발생해도 ACK나 NACK를 보내지 않고, 문제가 있는 프레임을 폐기한다.

```text
이더넷 -> 오류 검출 후 폐기
TCP    -> 손실 감지 및 재전송
```

---

## CSMA/CD

> CSMA/CD는 공유 매체에서 충돌을 감지하기 위한 MAC 프로토콜이다.

초기 이더넷은 브로드캐스트 링크를 사용했기 때문에 여러 노드가 동시에 전송하면 충돌이 발생할 수 있었다.  
이 충돌을 처리하기 위해 CSMA/CD가 사용되었다.

현재의 스위치 기반 이더넷은 각 호스트가 스위치와 점대점 링크로 연결된다.  
따라서 충돌이 거의 발생하지 않으며, CSMA/CD의 필요성은 크게 줄어들었다.

## 6.4.3 링크 계층 스위치

> 링크 계층 스위치는 들어온 프레임을 적절한 출력 인터페이스로 전달하는 장비이다.

스위치는 MAC 주소를 기반으로 동작한다.  
호스트와 라우터는 스위치가 중간에서 동작한다는 사실을 신경 쓰지 않아도 된다.

즉, 스위치는 호스트와 라우터에게 투명하게 동작한다.

---

## 포워딩과 필터링

스위치는 스위치 테이블을 이용해 프레임을 처리한다.

### 포워딩

> 프레임을 어떤 인터페이스로 내보낼지 결정하고 전달하는 기능이다.

### 필터링

> 프레임을 전달할지, 폐기할지 결정하는 기능이다.

---

## 스위치 테이블

> 스위치 테이블은 MAC 주소와 인터페이스 정보를 저장한다.

스위치 테이블에는 다음 정보가 들어간다.

- MAC 주소
- 해당 MAC 주소로 가기 위한 스위치 인터페이스
- 엔트리가 만들어진 시간

---

## 스위치 동작 방식

### 목적지 MAC 주소에 대한 엔트리가 없는 경우

스위치는 프레임을 수신한 인터페이스를 제외한 모든 인터페이스로 브로드캐스트한다.

### 목적지가 같은 인터페이스에 있는 경우

프레임을 다른 인터페이스로 보낼 필요가 없다.  
따라서 스위치는 프레임을 폐기한다.

이것이 필터링이다.

### 목적지가 다른 인터페이스에 있는 경우

스위치는 해당 인터페이스의 출력 버퍼로 프레임을 전달한다.

이것이 포워딩이다.

```text
모르면 브로드캐스트
같은 포트면 필터링
다른 포트면 포워딩
```

---

## 자가학습(Self-Learning)

> 스위치는 출발지 MAC 주소를 보고 스위치 테이블을 자동으로 만든다.

처음에는 스위치 테이블이 비어 있다.  
프레임이 들어오면 스위치는 출발지 MAC 주소와 들어온 인터페이스를 기록한다.

### 과정

1. 스위치 테이블은 초기에 비어 있다.
2. 프레임이 들어오면 출발지 MAC 주소를 확인한다.
3. 출발지 MAC 주소와 프레임이 들어온 인터페이스를 테이블에 저장한다.
4. 시간이 지나면서 여러 호스트의 위치를 자동으로 학습한다.
5. Aging Time이 지나도록 갱신되지 않은 엔트리는 삭제된다.

### 특징

- 자동 학습
- 동적 갱신
- 관리자 설정 불필요
- Plug-and-Play 장치

---

## 스위치의 특징

### 충돌 제거

스위치는 각 링크를 분리하기 때문에 충돌이 발생하지 않는다.  
또한 프레임을 버퍼링하고, 한 시점에 하나의 프레임만 세그먼트로 전송한다.

### 이질적인 링크 지원

스위치는 서로 다른 속도나 매체를 사용하는 링크들을 연결할 수 있다.

예:

- 꼬임쌍선
- 광섬유
- 서로 다른 전송 속도

### 관리 용이

스위치는 네트워크 관리를 쉽게 한다.

예:

- 링크 장애 감지
- 오동작 어댑터 차단
- 대역폭 사용 통계 수집
- 트래픽 종류 분석

---

## 스위치 보안

### 스위치 테이블 엔트리가 있는 경우

스위치는 프레임을 목적지 포트로만 전달한다.  
따라서 다른 호스트가 자신을 목적지로 하지 않는 프레임을 보기 어렵다.

### 스위치 테이블 엔트리가 없는 경우

스위치는 프레임을 브로드캐스트한다.  
이 경우 스니퍼가 자신을 목적지로 하지 않는 일부 프레임을 볼 수 있다.

### 스위치 포이즈닝(Switch Poisoning)

> 가짜 MAC 주소로 스위치 테이블을 오염시키는 공격이다.

공격자는 여러 가짜 출발지 MAC 주소를 가진 프레임을 많이 보낸다.  
그러면 스위치 테이블이 가짜 엔트리로 가득 차고, 정상적인 MAC 주소를 저장할 공간이 부족해진다.

그 결과 스위치가 더 많은 프레임을 브로드캐스트하게 되어 스니핑 가능성이 커질 수 있다.

---

## 스위치 vs 라우터

<p align="center">
  <img width="750" alt="스위치 대 라우터" src="https://user-images.githubusercontent.com/76640167/214864160-537f30f6-bca5-4b5a-a30f-f00fab1c9b5a.png">
</p>

| 구분 | 스위치 | 라우터 |
|---|---|---|
| 계층 | 2계층 | 3계층 |
| 사용 주소 | MAC 주소 | IP 주소 |
| 전달 단위 | 프레임 | 패킷 |
| 설정 | 비교적 자동 | 관리 설정 필요 |
| 특징 | 빠른 전달 | 경로 선택 가능 |

### 스위치의 장점

- Plug-and-Play 장치이다.
- 빠른 필터링과 포워딩이 가능하다.
- 충돌을 제거한다.
- 작은 네트워크에 적합하다.

### 스위치의 단점

- 브로드캐스트 폭풍에 취약할 수 있다.
- 대규모 네트워크에서는 ARP 트래픽이 증가한다.
- 루프 방지를 위해 스패닝 트리 제한이 필요하다.

### 라우터의 장점

- IP 주소 기반 계층 구조를 사용한다.
- 최적 경로 선택이 가능하다.
- 브로드캐스트 도메인을 분리할 수 있다.
- 브로드캐스트 폭풍을 막는 데 유리하다.
- 큰 네트워크에 적합하다.

### 라우터의 단점

- Plug-and-Play 장치가 아니다.
- 설정이 필요하다.
- 스위치보다 패킷 처리 시간이 더 크다.

---

## 6.4.4 VLAN

<p align="center">
  <img width="700" alt="가상 근거리 네트워크" src="https://user-images.githubusercontent.com/76640167/214864157-78d63e54-0306-4023-b37b-d6e98fa3babf.png">
</p>

> VLAN은 하나의 물리적 LAN에서 여러 개의 논리적 LAN을 구성하는 기술이다.

같은 스위치에 연결되어 있어도 VLAN이 다르면 서로 다른 LAN에 있는 것처럼 동작한다.  
VLAN을 사용하면 물리적 배선 구조는 유지하면서 논리적으로 네트워크를 분리할 수 있다.

---

## VLAN이 필요한 이유

기존의 계층적 스위치 구조만으로는 다음 문제가 생길 수 있다.

### 트래픽 격리 부족

브로드캐스트 트래픽이 전체 네트워크로 전달될 수 있다.  
브로드캐스트 범위를 제한하면 LAN 성능을 향상할 수 있다.

### 스위치의 비효율적 사용

그룹마다 별도 스위치를 사용하면 포트가 낭비될 수 있다.  
하나의 큰 스위치로 모든 호스트를 연결할 수는 있지만, VLAN이 없으면 트래픽 격리가 어렵다.

### 사용자 관리 문제

사용자가 다른 그룹으로 이동하면 물리적 케이블 연결을 바꿔야 할 수 있다.  
VLAN을 사용하면 포트 설정만 변경해도 그룹 이동을 처리할 수 있다.

---

## 포트 기반 VLAN

<p align="center">
  <img width="700" alt="기본 VLAN" src="https://user-images.githubusercontent.com/76640167/214864149-ba81a944-ce7b-4f0b-9942-21bbae0fe277.png">
</p>

> 포트 기반 VLAN은 스위치 포트를 그룹으로 나누어 VLAN을 구성하는 방식이다.

같은 VLAN에 속한 포트들은 하나의 브로드캐스트 도메인을 형성한다.  
따라서 한 VLAN에서 발생한 브로드캐스트 프레임은 같은 VLAN 안에서만 전달된다.

예:

```text
포트 2~8  -> EE VLAN
포트 9~15 -> CS VLAN
```

VLAN 스위치는 포트와 VLAN의 매핑 테이블을 관리한다.  
관리자는 스위치 관리 소프트웨어를 통해 포트가 속한 VLAN을 변경할 수 있다.

---

## VLAN 간 통신

> 서로 다른 VLAN은 논리적으로 다른 LAN이므로 직접 통신할 수 없다.

서로 다른 VLAN 사이에서 통신하려면 라우터나 3계층 스위치가 필요하다.

```text
VLAN A 호스트
-> 라우터 또는 3계층 스위치
-> VLAN B 호스트
```

즉, VLAN은 브로드캐스트 도메인을 분리하지만, VLAN 간 통신을 위해서는 네트워크 계층 장비가 필요하다.

---

## 외부 스위치와 VLAN 연결

<p align="center">
  <img width="700" alt="VLAN 외부 스위치 연결" src="https://user-images.githubusercontent.com/76640167/214864142-d61da77a-1d2e-4769-b60c-513ec4e23c86.png">
</p>

외부 스위치가 VLAN에 속하려면 각 VLAN마다 별도의 연결 포트가 필요할 수 있다.  
하지만 VLAN 수가 많아지면 필요한 포트 수도 증가한다.

```text
VLAN N개 -> 연결 포트 N개 필요
```

이 방식은 VLAN 수가 늘어날수록 확장성이 떨어진다.

---

## VLAN 트렁킹

<p align="center">
  <img width="700" alt="VLAN 트렁킹" src="https://user-images.githubusercontent.com/76640167/214864132-da03841d-5eed-4dd0-a262-43be473ea3e1.png">
</p>

> VLAN 트렁킹은 여러 VLAN의 프레임을 하나의 링크로 전달하는 방식이다.

트렁크 포트는 여러 VLAN의 트래픽을 함께 전달할 수 있다.  
따라서 스위치 간 연결에서 VLAN마다 별도의 포트를 사용할 필요가 없다.

### 트렁크 포트의 특징

- 여러 VLAN의 프레임을 전달할 수 있다.
- 스위치 간 연결에 사용된다.
- VLAN마다 별도 링크를 두지 않아도 된다.
- 모든 VLAN에 속한 포트처럼 동작한다.

---

## VLAN 태그

<p align="center">
  <img width="700" alt="VLAN 태그" src="https://user-images.githubusercontent.com/76640167/214872229-68994ade-4e37-4c1c-9f9c-84195afe7de8.png">
</p>

> VLAN 태그는 프레임이 어느 VLAN에 속하는지 표시하는 정보이다.

트렁크 링크를 지나는 프레임에는 여러 VLAN의 트래픽이 섞여 있다.  
따라서 스위치는 VLAN 태그를 사용해 프레임의 VLAN 소속을 구분한다.

### 특징

- 송신 측 스위치가 VLAN 태그를 추가한다.
- 수신 측 스위치가 VLAN 태그를 해석한다.
- 필요한 경우 수신 측 스위치가 태그를 제거한다.

### VLAN 태그 구성

VLAN 태그는 4바이트 크기이다.

- TPID: Tag Protocol Identifier
- TCI: Tag Control Information
- VLAN ID: VLAN 식별자
- Priority: 우선순위 정보

VLAN ID를 통해 스위치는 프레임이 어느 VLAN에 속하는지 알 수 있다.
