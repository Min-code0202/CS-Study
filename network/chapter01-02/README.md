## 쿠키

쿠키(cookie)는 브라우저가 저장해두는 작은 정보 조각이다.

HTTP는 무상태(stateless) 프로토콜이기 때문에 서버는 이전 요청을 기억하지 않는다. 따라서 사용자를 계속 식별해야 하는 기능에서는 요청할 때마다 사용자가 누구인지 알 수 있는 정보가 함께 전달되어야 한다.

### 쿠키가 필요한 이유

예를 들어 사용자가 쇼핑몰을 이용한다고 하면 다음 요청들이 순서대로 발생할 수 있다.

1. 로그인 요청
2. 상품 페이지 요청
3. 장바구니 요청
4. 결제 페이지 요청

HTTP 자체만 보면 서버는 각 요청을 독립적인 요청으로 본다.

```text
요청 1: 누구세요?
요청 2: 누구세요?
요청 3: 누구세요?
요청 4: 누구세요?
```

그래서 요청할 때마다 사용자를 식별할 수 있는 정보가 필요하다. 쿠키는 이 정보를 브라우저가 저장하고, 이후 같은 서버로 요청할 때 자동으로 함께 보내도록 해준다.

### 쿠키 동작 과정

1. 사용자가 로그인 요청을 보낸다.
2. 서버가 사용자를 확인한다.
3. 서버가 응답하면서 쿠키를 심어준다.
4. 브라우저는 쿠키를 쿠키 저장소에 저장한다.
5. 이후 요청부터 브라우저가 쿠키를 자동으로 보낸다.

중요한 점은 쿠키에 사용자 정보를 직접 담지 않는 것이다. 보통 쿠키에는 `sessionId` 같은 식별자만 담고, 실제 사용자 정보는 서버 쪽 저장소에서 관리한다.

### 예시

서버는 브라우저에게 다음과 같은 쿠키를 내려줄 수 있다.

```http
Set-Cookie: sessionId=abc123
```

서버는 DB나 Redis 같은 세션 저장소에 다음과 같은 정보를 저장한다.

```text
sessionId: abc123
userId: junseok
expiresAt: 2026-04-28 15:00
```

이후 브라우저가 같은 서버로 요청할 때 `sessionId=abc123` 쿠키를 자동으로 보내면, 서버는 그 값을 이용해 세션 저장소에서 사용자를 찾는다.

### 쿠키 전송 범위

쿠키는 아무 서버에나 보내지 않는다.

예를 들어 `example.com`에 대한 쿠키는 기본적으로 `example.com`으로 요청할 때만 전송된다. 쿠키의 전송 범위는 `Domain`, `Path` 같은 속성으로 조절할 수 있다.

### 쿠키 속성

- `Domain`: 쿠키를 전송할 도메인을 정한다.
- `Path`: 쿠키를 전송할 경로 범위를 정한다.
- `Expires`: 쿠키의 만료 시각을 정한다.
- `HttpOnly`: JavaScript에서 쿠키에 접근하지 못하게 한다.
- `Secure`: HTTPS 요청에서만 쿠키를 전송하게 한다.
- `SameSite`: 다른 사이트에서 출발한 요청에 쿠키를 보낼지 제한한다.

## 쿠키와 세션

### 쿠키

쿠키는 브라우저에 저장된다. 브라우저는 요청할 때 쿠키를 자동으로 서버에 전송한다.

### 세션

세션은 서버가 사용자의 로그인 상태 등을 저장하는 공간이다. 서버는 쿠키에 들어있는 `sessionId`를 이용해 세션 저장소에서 사용자의 상태를 찾는다.

즉, 쿠키는 브라우저가 들고 있는 식별자이고, 세션은 서버가 관리하는 사용자 상태 정보다.

### 쿠키와 세션의 관계

```text
브라우저 쿠키
sessionId=abc123

서버 세션 저장소
abc123 -> userId: junseok
```

서버는 세션 저장소를 관리해야 한다. 세션 저장소는 DB나 Redis에 둘 수 있다.

같은 사용자가 여러 기기에서 로그인하면 세션이 여러 개 생길 수 있다.

```text
노트북 로그인 -> sessionId=A
휴대폰 로그인 -> sessionId=B
태블릿 로그인 -> sessionId=C
```

각 기기는 서로 다른 쿠키를 가질 수 있고, 서버는 각각의 `sessionId`에 대응하는 세션을 별도로 관리한다.
