## 2.1 네트워크 애플리케이션의 원리

클라이언트-서버 구조 : 항상 켜진 서버 + 요청하는 클라이언트. 클라이언트끼리는 직접 통신 안 함.

P2P 구조 : 중앙 서버 없이 피어끼리 직접 통신. 자가 확장성(self-scalability) 보유.

소켓(socket) : 애플리케이션 계층과 트랜스포트 계층 사이의 인터페이스. 프로세스가 네트워크와 통신하는 문.

프로세스 주소 = IP 주소 + 포트 번호(port number).

TCP : 연결 지향형. 신뢰적 전송 + 흐름 제어 + 혼잡 제어 제공. 핸드셰이킹으로 연결 수립.

UDP : 비연결형. 위 기능 없음. 빠름. 스트리밍처럼 속도가 중요한 곳에 사용.

TLS(Transport Layer Security) : 기본 TCP에 암호화를 추가한 것. HTTPS가 사용.

> 면접 포인트 

**TCP vs UDP 차이**
TCP는 연결을 먼저 맺고(핸드셰이킹) 데이터를 전송. 전달 보장, 순서 보장, 흐름 제어, 혼잡 제어를 제공하는 대신 느림. UDP는 연결 없이 바로 전송. 위 기능 없음. 빠름. 신뢰성보다 속도가 중요한 실시간 스트리밍, DNS, 게임 등에 사용.

**소켓이 뭔지**
프로세스가 네트워크와 통신하기 위한 인터페이스. 애플리케이션 계층과 트랜스포트 계층 사이의 문. 개발자는 소켓을 통해 데이터를 보내고 받으며, 내부적으로 어떻게 전송되는지는 트랜스포트 계층이 처리.

**클라이언트-서버 vs P2P 구조 차이**
클라이언트-서버는 서버가 모든 요청을 처리하기 때문에 사용자가 늘수록 서버 부하가 선형으로 증가. P2P는 피어가 늘수록 요청자도 늘지만 공급자도 늘어서 부하가 분산됨. 그래서 자가 확장성을 가진다고 함.

**TCP의 3가지 서비스**
연결 지향형(핸드셰이킹으로 연결 수립), 신뢰적 데이터 전송(손실/순서 보장), 혼잡 제어(네트워크 혼잡 시 속도 조절).

## 2.2 웹과 HTTP

HTTP : TCP 기반. 비상태(stateless) 프로토콜. 서버는 클라이언트의 이전 요청을 기억하지 않음.

비지속 연결 : 요청마다 TCP 연결 새로 맺음. 객체 하나당 2RTT 소요.

지속 연결 : TCP 연결 유지. 파이프라이닝(pipelining)으로 응답 기다리지 않고 연속 요청 가능. HTTP/1.1 기본.

주요 HTTP 메서드 : GET, POST, PUT, DELETE, HEAD.

주요 상태 코드 : 200 OK, 301 Moved Permanently, 400 Bad Request, 404 Not Found, 505 Version Not Supported.

쿠키(cookie) : stateless인 HTTP에서 사용자를 식별하기 위한 수단. Set-cookie → 저장 → Cookie 헤더로 재전송.

웹 캐시(proxy server) : 기점 서버 대신 응답. 응답 시간 단축 + 트래픽 감소.

조건부 GET : If-modified-since 헤더로 캐시 유효성 확인. 변경 없으면 304 Not Modified 반환.

HTTP/2 : HOL 블로킹 문제를 프레이밍(framing)으로 해결. 메시지를 프레임으로 쪼개 인터리빙 전송. 하나의 TCP 연결로 처리.

> 면접 포인트

**HTTP stateless가 뭔지**
서버가 클라이언트의 이전 요청 정보를 저장하지 않는다는 뜻. 같은 사용자가 두 번 요청해도 서버는 동일인인지 모름. 덕분에 서버 설계가 단순해지지만 상태 유지가 필요한 경우(로그인 등)엔 쿠키나 세션으로 보완해야 함.

**GET vs POST**
GET은 데이터를 URL에 담아 요청. 조회 목적. 캐싱 가능. POST는 데이터를 body에 담아 전송. 생성/수정 목적. 캐싱 안 됨. 보안상 민감한 데이터는 POST 사용.

**상태 코드**
2xx는 성공, 3xx는 리다이렉션, 4xx는 클라이언트 오류, 5xx는 서버 오류. 자주 나오는 건 200(성공), 201(생성됨), 301(영구 이동), 400(잘못된 요청), 401(인증 필요), 403(권한 없음), 404(없음), 500(서버 오류).

**쿠키 동작 방식**
서버가 응답할 때 Set-Cookie 헤더에 식별값을 담아 보냄 → 브라우저가 저장 → 이후 같은 서버에 요청할 때마다 Cookie 헤더에 담아서 전송 → 서버가 사용자 식별. stateless인 HTTP에서 상태를 유지하는 방법.

**웹 캐시와 조건부 GET**
웹 캐시는 자주 요청되는 객체를 저장해두고 기점 서버 대신 응답해서 응답 시간을 줄임. 문제는 캐시가 오래됐을 수 있다는 것. 조건부 GET으로 해결. If-modified-since 헤더에 마지막으로 받은 시간을 담아 보내면 서버가 그 이후 변경이 없으면 304만 보냄. 불필요한 데이터 전송 없이 캐시 유효성 확인 가능.

**HTTP/1.1 vs HTTP/2 차이**
HTTP/1.1은 하나의 TCP 연결에서 요청/응답이 순서대로 처리됨. 큰 파일이 앞에 있으면 뒤 요청이 밀리는 HOL(Head of Line) 블로킹 문제 발생. HTTP/2는 메시지를 프레임 단위로 쪼개서 동시에 여러 요청/응답을 인터리빙하여 처리. HOL 블로킹 해결. 헤더 압축, 서버 푸시도 추가됨.

## 2.3 인터넷 전자메일

SMTP : 메일 서버 간 전송 프로토콜. TCP 포트 25번. push 방식. 7-bit ASCII만 허용.

메일 접근은 pull 방식이라 별도 프로토콜 필요. HTTP(웹메일) 또는 IMAP 사용.

전송 흐름 : 송신자 UA → 송신자 메일 서버(SMTP) → 수신자 메일 서버(SMTP) → 수신자 UA(HTTP or IMAP).

> 면접 포인트

**SMTP가 뭔지**
Simple Mail Transfer Protocol. 메일 서버 간에 메일을 전송하는 프로토콜. TCP 포트 25번 사용. 송신자 메일 서버에서 수신자 메일 서버로 직접 전달(중간 서버 경유 없음). 전송 실패 시 메시지는 송신자 메일 서버에 남아있고 30분마다 재시도.

**push/pull 개념**
SMTP는 push. 송신자가 수신자 메일 서버로 밀어 넣는 방식. 반면 수신자가 메일 서버에서 메일을 가져오는 건 pull. HTTP나 IMAP 같은 별도 프로토콜 필요. SMTP 하나로 송수신을 다 처리할 수 없는 이유가 여기에 있음.

**송수신 흐름**
송신자가 메일 작성 → 송신자 UA가 송신자 메일 서버로 전송(SMTP 또는 HTTP) → 송신자 메일 서버가 수신자 메일 서버로 전송(SMTP) → 수신자가 메일 서버에서 가져옴(HTTP 또는 IMAP).

## 2.4 DNS

DNS(Domain Name System) : 호스트 이름 → IP 주소 변환. 분산 계층 데이터베이스. UDP 포트 53번.

계층 구조 : 루트 → TLD → 책임(authoritative) → 로컬 DNS 서버.

IP 찾는 방식 : 호스트 → 로컬 DNS는 "대신 알아봐줘" 방식(재귀적). 로컬 DNS → 루트/TLD/책임 서버는 "다음 서버 주소만 알려줘" 방식(반복적). 루트 서버 부하를 줄이기 위해 반복적 방식이 일반적.

edu.ssafy.com IP 찾는 과정 : 로컬 → 루트(com TLD 주소 반환) → TLD(ssafy.com 책임 서버 주소 반환) → 책임 서버(최종 IP 반환) → 로컬 → 호스트. 총 8번 메시지 교환.

DNS 캐싱 : 응답을 로컬에 저장. TTL 이후 제거. 루트 서버 우회 가능.

자원 레코드 타입 : A(호스트→IP), NS(도메인→책임 서버), CNAME(별칭→정식 이름), MX(메일 서버).

> 면접 포인트

**DNS가 뭔지**
사람이 읽기 쉬운 호스트 이름(domain)을 라우터가 처리할 수 있는 IP 주소로 변환해주는 시스템. 전 세계에 분산된 계층 구조 데이터베이스로 운영. UDP 포트 53번 사용.

**계층 구조**
루트 서버가 최상위. TLD 서버(.com, .kr 등)가 그 아래. 각 도메인의 책임 서버가 실제 IP 정보를 보유. 로컬 DNS 서버는 ISP가 운영하며 사용자와 가장 가까운 위치에서 캐시 역할도 함.

**IP 찾는 과정 (브라우저에 URL 입력 시)**
브라우저가 로컬 DNS에 요청 → 캐시에 없으면 루트 서버에 물어봄 → 루트가 TLD 서버 주소 알려줌 → TLD 서버가 책임 서버 주소 알려줌 → 책임 서버가 최종 IP 반환 → 로컬 DNS가 캐싱 후 브라우저에 전달 → 브라우저가 해당 IP로 TCP 연결 후 HTTP 요청.

**캐싱**
한 번 찾은 IP는 TTL 동안 로컬에 저장. 같은 도메인 재요청 시 루트/TLD까지 안 올라가도 됨. TTL 지나면 삭제되고 다시 탐색.

**URL 입력 시 일어나는 일**
브라우저 주소창에 입력 → DNS로 IP 조회 → 해당 IP의 80번(HTTP) 또는 443번(HTTPS) 포트로 TCP 연결 → HTTP 요청 전송 → 서버가 응답 → 브라우저가 렌더링.

## 2.5 P2P 파일 분배

클라이언트-서버 분배 시간 D(cs) ≥ max{ NF/u(s), F/d(min) } → N에 따라 선형 증가.

P2P 분배 시간 D(p2p) ≥ max{ F/u(s), F/d(min), NF/u(total) } → 피어 증가 시 u(total)도 증가해서 완만하게 유지. 자가 확장성.

비트토렌트 : rarest first(희귀한 청크 먼저 요청), tit-for-tat(빠르게 보내주는 피어에게 우선 전송).

> 면접 포인트

**클라이언트-서버 vs P2P 확장성 차이**
클라이언트-서버는 사용자가 N명이면 서버가 NF 비트를 전부 전송해야 함. N이 늘수록 서버 부하가 선형 증가. P2P는 피어가 파일을 받으면서 동시에 다른 피어에게 나눠줌. 피어가 늘수록 전체 업로드 용량(u(total))도 늘어서 분배 시간이 완만하게 유지됨. 이게 자가 확장성.

## 2.6 비디오 스트리밍과 CDN

DASH : 비디오를 여러 화질 버전으로 만들어두고, 클라이언트가 현재 대역폭에 맞는 버전을 골라서 요청. 매니페스트(manifest) 파일로 버전 목록 확인 후 chunk 단위로 요청.

CDN : 콘텐츠 복사본을 전 세계 분산 서버에 저장. DNS redirection으로 가까운 서버로 연결.

Enter Deep : 사용자 근처에 서버 최대한 많이 배치. 지연 적지만 유지비 큼.

Bring Home : 핵심 거점(IXP)에 큰 클러스터 몇 개만 운영. 유지비 적지만 지연 더 있을 수 있음.

> 면접 포인트

**DASH가 뭔지**
Dynamic Adaptive Streaming over HTTP. 네트워크 상황에 따라 비디오 화질을 동적으로 조정하는 방식. 비디오를 여러 화질 버전으로 인코딩해두고, 클라이언트가 현재 대역폭을 보고 알맞은 버전을 chunk 단위로 요청. 대역폭이 줄면 자동으로 낮은 화질로 전환. 유튜브, 넷플릭스 다 이 방식.

**CDN이 왜 필요한지**
단일 서버로 전 세계에 스트리밍하면 먼 사용자는 지연이 크고, 같은 콘텐츠를 반복 전송해야 해서 비용 낭비. 서버 하나가 죽으면 전체 서비스 중단. CDN은 콘텐츠를 전 세계 분산 서버에 복사해두고 사용자와 가까운 서버에서 제공해서 이 문제들을 해결.
