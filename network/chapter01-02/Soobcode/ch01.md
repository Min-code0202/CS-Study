# 1단원 핵심 정리

---

## 1.1 인터넷 구성요소

호스트(host / end system) : 인터넷에 연결된 기기. 데이터를 보내고 받는 주체.

패킷 스위치(packet switch) : 패킷의 경로를 설정. 라우터(router)와 링크 계층 스위치(link-layer switch)가 대표적.

통신 링크(communication link) : 광케이블, 구리선, 와이파이 등 데이터를 물리적으로 전달하는 매체. 각 링크는 전송률(transmission rate, bps)을 가짐.

ISP(Internet Service Provider) : 인터넷 연결을 제공하는 회사. 호스트들은 ISP를 통해 인터넷에 접속. 여러 ISP들이 서로 연결되어 전체 인터넷을 구성함. 인터넷 = 네트워크들의 네트워크(network of networks).

프로토콜(protocol) : 둘 이상의 개체가 데이터를 주고받기 위해 약속된 규칙. 포맷, 순서, 동작으로 구성. ex) HTTP, TCP, IP, WiFi.

> 면접 포인트

**인터넷이 뭔지**
인터넷은 전 세계 수많은 네트워크들이 ISP를 통해 서로 연결된 "네트워크들의 네트워크". 호스트들이 패킷 스위치와 통신 링크로 구성된 네트워크를 통해 서로 데이터를 주고받는 시스템.

**프로토콜이 뭔지**
통신하는 두 개체가 데이터를 주고받기 위해 미리 약속한 규칙. 어떤 형식으로 메시지를 보낼지, 어떤 순서로 주고받을지, 어떤 상황에서 어떻게 행동할지를 정의함. ex) HTTP는 웹 브라우저와 서버가 통신하는 규칙.

## 1.2 접속 네트워크

접속 네트워크(access network) : 호스트를 첫 번째 라우터(가장자리 라우터)에 연결하는 네트워크.

가정 접속 : DSL(전화선), 케이블(케이블TV), FTTH(광섬유), 5G.

기업/광역 접속 : 이더넷(꼬임쌍선+스위치), WiFi(IEEE 802.11), LTE/5G.

공유 매체 : 케이블처럼 여러 사용자가 대역폭을 나눠 쓰는 구조. 사용자 많을수록 속도 저하.

> 면접 포인트

**대역폭(bandwidth)이 뭔지**
링크가 처리할 수 있는 최대 데이터 전송 용량. 단위는 bps(bits per second). 대역폭이 클수록 단위 시간에 더 많은 데이터를 전송 가능. 요즘은 전송률(transmission rate)과 같은 의미로 혼용.

---

## 1.3 네트워크 코어

### 패킷 교환 vs 회선 교환

패킷 교환 : 자원 예약 없이 필요할 때만 링크 사용. 전송 보장 없음. 인터넷이 사용하는 방식.

회선 교환 : 통신 전에 경로상 자원을 미리 예약. 일정 전송률 보장. 전화망이 사용하는 방식. 비활용 기간에도 자원 점유해서 낭비 발생.

패킷 교환이 더 효율적인 이유 : 사용자가 항상 데이터를 보내지 않기 때문. 100명 중 실제로 동시에 전송하는 사람은 극히 일부라서 자원을 공유해도 충분히 처리 가능.

### 저장-후-전달 (Store and Forward)

라우터는 패킷을 전부 받아야 다음으로 전달. 종단 간 지연 = N × L/R (N : 링크 수, L : 패킷 크기, R : 전송률).

### 큐잉 지연 & 패킷 손실

링크가 바쁘면 출력 버퍼에서 대기(큐잉 지연). 버퍼가 꽉 차면 패킷을 버림(패킷 손실). 혼잡할수록 심해짐.

### ISP 계층 구조

1계층 ISP(글로벌) → 지역 ISP → 접속 ISP → 사용자 순서로 계층화.

PoP : 제공자 네트워크의 라우터 그룹. 고객 ISP가 여기에 연결.

멀티홈(multi-homing) : ISP 2개 이상에 연결. 하나가 죽어도 인터넷 유지.

피어링(peering) : 같은 계층 ISP끼리 직접 연결. 상위 ISP 거치지 않아 비용 절감.

IXP : 여러 ISP가 피어링하는 물리적 장소.

> 면접 포인트

**패킷 교환 vs 회선 교환 차이**
회선 교환은 통신 전에 경로상 자원을 미리 예약하고 통신이 끝날 때까지 점유. 전송률이 보장되지만 비활용 시간에도 자원이 낭비됨. 패킷 교환은 자원을 미리 예약하지 않고 필요할 때만 사용. 자원 효율이 높지만 혼잡 시 지연이나 손실이 발생할 수 있음. 인터넷은 패킷 교환 사용.

**라우터가 하는 일**
패킷이 도착하면 헤더의 목적지 IP를 확인하고 포워딩 테이블을 참조해서 어느 링크로 내보낼지 결정. 포워딩 테이블은 라우팅 프로토콜이 자동으로 설정.

## 1.4 지연, 손실, 처리율

### 전체 노드 지연 = 처리 지연 + 큐잉 지연 + 전송 지연 + 전파 지연

처리 지연 : 헤더 보고 경로 결정하는 시간. 보통 무시 가능.

큐잉 지연 : 버퍼에서 기다리는 시간. 혼잡도에 따라 크게 달라짐.

전송 지연 : 패킷을 링크에 다 올리는 시간. L/R. 거리와 무관.

전파 지연 : 비트가 링크를 물리적으로 건너는 시간. d/s. 패킷 크기/전송률과 무관.

전송 지연 vs 전파 지연 : 전송 지연은 패킷을 링크에 올리는 시간, 전파 지연은 이미 올라간 비트가 목적지까지 이동하는 시간. 자주 헷갈리는 개념.

### 트래픽 강도 (La/R)

La/R > 1 : 큐 무한 증가 → 시스템 마비. 이렇게 설계하면 안 됨.

La/R → 1 : 큐잉 지연 급격히 증가.

La/R → 0 : 큐잉 지연 거의 없음.

### 처리율 & 병목 링크

처리율 = 경로상 가장 느린 링크(병목 링크)의 전송률. min{Rs, Rc}.

오늘날 병목은 주로 접속 네트워크(Rs 또는 Rc)에서 발생. 코어 링크는 충분히 빠름.

> 면접 포인트

**지연의 4가지 종류**
처리 지연(헤더 보고 경로 결정), 큐잉 지연(버퍼 대기), 전송 지연(패킷을 링크에 올리는 시간, L/R), 전파 지연(비트가 링크를 건너는 시간, d/s). 전송 지연은 패킷 크기와 전송률에 의존하고 거리와 무관. 전파 지연은 거리에 의존하고 패킷 크기와 무관.

**병목 링크가 뭔지**
경로상에서 처리율을 제한하는 가장 느린 링크. 전체 처리율은 병목 링크의 전송률로 결정됨. 아무리 다른 링크가 빨라도 병목 링크 속도 이상으로 데이터를 전달할 수 없음.

## 1.5 프로토콜 계층

### 인터넷 프로토콜 스택 (5계층)

5계층 애플리케이션(Application) : HTTP, SMTP, FTP, DNS. 데이터 단위는 메시지(message).

4계층 트랜스포트(Transport) : TCP, UDP. 데이터 단위는 세그먼트(segment).

3계층 네트워크(Network) : IP. 데이터 단위는 데이터그램(datagram). 출발지→목적지 라우팅 담당.

2계층 링크(Link) : 이웃 노드 간 전달. 데이터 단위는 프레임(frame).

1계층 물리(Physical) : 비트를 실제 신호로 전송. 전송 매체에 의존.

### 각 장치가 구현하는 계층

호스트 : 5계층 전부. 라우터 : 1~3계층. 링크 계층 스위치 : 1~2계층.

### 캡슐화 (Encapsulation)

각 계층이 상위 계층 패킷을 페이로드로 삼고 자기 헤더를 붙여서 내려보냄.

메시지(M) → 세그먼트(Ht+M) → 데이터그램(Hn+Ht+M) → 프레임(Hl+Hn+Ht+M).

수신 측에서는 반대로 각 계층이 자기 헤더를 떼어내면서 위로 올려보냄.

> 면접 포인트
(참고)
**OSI 7계층 vs TCP/IP 5계층**
OSI는 이론적 표준 모델로 7계층(물리, 데이터링크, 네트워크, 트랜스포트, 세션, 표현, 애플리케이션). TCP/IP는 실제 인터넷에서 사용하는 5계층 모델로 세션/표현 계층이 애플리케이션 계층에 합쳐짐. 실무에서는 TCP/IP 기준으로 이야기하는 경우가 많음.

**캡슐화가 뭔지**
데이터를 전송할 때 각 계층이 자기 계층의 헤더를 붙여 아래로 내려보내는 것. 수신 측에서는 반대로 헤더를 벗겨내면서 위로 올려보냄. 덕분에 각 계층은 자기 헤더만 신경 쓰면 되고 다른 계층의 내부 동작은 몰라도 됨. 계층 구조의 핵심 장점.

**각 계층의 역할과 대표 프로토콜**
애플리케이션(HTTP, DNS, SMTP) → 트랜스포트(TCP, UDP) → 네트워크(IP) → 링크(이더넷, WiFi) → 물리(광케이블, 구리선). 면접에서 "HTTP는 몇 계층?" 같은 질문 자주 나옴. HTTP는 애플리케이션 계층(5계층), TCP는 트랜스포트 계층(4계층), IP는 네트워크 계층(3계층).
