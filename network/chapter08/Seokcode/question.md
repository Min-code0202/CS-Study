# Network 면접 질문 정리

## 1. 인터넷에서 패킷 교환 방식과 회선 교환 방식의 차이는 무엇인가요?

패킷 교환은 데이터를 작은 패킷 단위로 나누어 네트워크 자원을 필요할 때마다 공유해서 사용하는 방식입니다. 각 패킷은 라우터를 거쳐 목적지까지 전달되며, 혼잡이 발생하면 큐잉 지연이나 패킷 손실이 생길 수 있습니다.

회선 교환은 통신을 시작하기 전에 송신자와 수신자 사이에 전용 경로와 자원을 미리 예약하는 방식입니다. 연결 중에는 일정한 대역폭이 보장되지만, 실제 데이터를 보내지 않는 시간에도 자원이 점유되어 비효율적일 수 있습니다.

## 2. 네트워크 지연을 구성하는 처리 지연, 큐잉 지연, 전송 지연, 전파 지연은 각각 어떤 의미인가요?

처리 지연은 라우터가 패킷 헤더를 검사하고 어디로 보낼지 결정하는 데 걸리는 시간입니다. 큐잉 지연은 패킷이 출력 링크로 나가기 전 버퍼에서 기다리는 시간입니다.

전송 지연은 패킷의 모든 비트를 링크에 밀어 넣는 데 걸리는 시간으로, `패킷 길이 / 전송률`로 계산합니다. 전파 지연은 비트가 물리 매체를 따라 다음 노드까지 이동하는 데 걸리는 시간입니다.

## 3. OSI/TCP-IP 계층 구조에서 애플리케이션 계층, 트랜스포트 계층, 네트워크 계층, 링크 계층은 각각 어떤 역할을 하나요?

애플리케이션 계층은 HTTP, DNS, SMTP처럼 사용자 애플리케이션이 사용하는 통신 기능을 제공합니다. 트랜스포트 계층은 TCP, UDP를 통해 프로세스 간 통신을 담당합니다.

네트워크 계층은 IP를 기반으로 호스트 간 패킷 전달과 라우팅을 담당합니다. 링크 계층은 같은 링크나 같은 네트워크 구간에서 프레임을 전달하며, Ethernet, Wi-Fi, ARP 등이 관련됩니다.

## 4. TCP와 UDP의 차이를 설명하고, 각각 어떤 상황에서 사용하는 것이 적절한가요?

TCP는 연결 지향형 프로토콜이며 신뢰성 있는 전송, 순서 보장, 흐름 제어, 혼잡 제어를 제공합니다. 웹, 이메일, 파일 전송처럼 데이터가 정확히 도착해야 하는 경우에 적합합니다.

UDP는 비연결형 프로토콜이며 전송 보장이나 순서 보장을 제공하지 않습니다. 대신 오버헤드가 작고 빠르기 때문에 DNS, 실시간 스트리밍, 온라인 게임처럼 속도나 지연이 중요한 경우에 사용됩니다.

## 5. HTTP가 stateless 프로토콜이라는 것은 어떤 의미이며, 쿠키는 이 한계를 어떻게 보완하나요?

HTTP가 stateless라는 것은 서버가 이전 요청의 상태를 기본적으로 기억하지 않는다는 의미입니다. 즉, 같은 클라이언트가 여러 번 요청하더라도 HTTP 자체만으로는 같은 사용자라는 사실을 알 수 없습니다.

쿠키는 서버가 `Set-Cookie` 헤더로 클라이언트에게 식별 정보를 저장하게 하고, 이후 요청에서 클라이언트가 `Cookie` 헤더로 다시 보내게 합니다. 이를 통해 로그인 상태, 장바구니, 세션 식별 같은 상태를 유지할 수 있습니다.

## 6. 비지속 HTTP와 지속 HTTP의 차이는 무엇이고, 지속 연결을 사용하면 어떤 장점이 있나요?

비지속 HTTP는 객체 하나를 요청하고 응답받을 때마다 TCP 연결을 새로 만들고 닫는 방식입니다. 객체마다 TCP 연결 설정 비용이 발생하므로 RTT와 자원 사용이 증가합니다.

지속 HTTP는 하나의 TCP 연결을 유지하면서 여러 HTTP 요청과 응답을 처리하는 방식입니다. 연결 설정 비용을 줄일 수 있고, 여러 객체를 더 효율적으로 가져올 수 있습니다.

## 7. DNS는 어떤 역할을 하며, 일반적으로 UDP를 사용하는 이유는 무엇인가요?

DNS는 사람이 읽기 쉬운 도메인 이름을 IP 주소로 변환하는 시스템입니다. 예를 들어 `example.com`을 실제 통신에 필요한 IP 주소로 바꿔줍니다.

DNS는 보통 요청과 응답 크기가 작고, 빠른 응답이 중요하기 때문에 UDP를 사용합니다. UDP는 TCP처럼 3-way handshake가 필요 없어서 연결 설정 지연이 없습니다. 다만 응답이 크거나 신뢰성이 필요한 경우 TCP를 사용할 수도 있습니다.

## 8. TCP의 3-way handshake 과정과 각 단계의 목적을 설명해주세요.

TCP 3-way handshake는 클라이언트와 서버가 연결을 설정하기 위한 과정입니다. 먼저 클라이언트가 서버에게 `SYN`을 보내 연결을 요청합니다.

서버는 요청을 수락하며 `SYN + ACK`를 보냅니다. 마지막으로 클라이언트가 `ACK`를 보내면 양쪽 모두 연결이 성립되었다고 판단하고 데이터를 주고받을 수 있습니다. 이 과정에서 초기 시퀀스 번호도 교환됩니다.

## 9. TCP에서 신뢰성 있는 데이터 전송을 위해 시퀀스 번호, ACK, 재전송 타이머가 어떻게 사용되나요?

TCP는 바이트 스트림에 시퀀스 번호를 붙여 데이터의 순서를 관리합니다. 수신자는 정상적으로 받은 데이터의 다음 바이트 번호를 ACK로 알려줍니다.

송신자는 ACK를 받지 못하면 데이터가 손실되었다고 판단할 수 있습니다. 이때 재전송 타이머가 만료되거나 중복 ACK가 반복되면 해당 세그먼트를 다시 전송하여 신뢰성을 보장합니다.

## 10. TCP의 흐름 제어와 혼잡 제어는 각각 어떤 문제를 해결하기 위한 기능인가요?

흐름 제어는 송신자가 수신자의 처리 능력이나 버퍼 크기보다 더 빠르게 데이터를 보내는 문제를 막기 위한 기능입니다. 수신자는 receive window를 통해 자신이 받을 수 있는 양을 송신자에게 알려줍니다.

혼잡 제어는 네트워크 내부의 라우터나 링크가 과부하되는 문제를 줄이기 위한 기능입니다. TCP는 `cwnd`를 조절하며 slow start, congestion avoidance, fast recovery 같은 방식으로 전송 속도를 조정합니다.

## 11. 라우터에서 포워딩과 라우팅의 차이는 무엇인가요?

포워딩은 라우터가 들어온 패킷을 포워딩 테이블을 보고 적절한 출력 포트로 내보내는 동작입니다. 개별 라우터 내부에서 빠르게 수행되는 데이터 평면의 기능입니다.

라우팅은 출발지에서 목적지까지 어떤 경로로 패킷을 보낼지 결정하는 과정입니다. 라우팅 알고리즘이나 프로토콜을 통해 포워딩 테이블을 만드는 제어 평면의 기능입니다.

## 12. 링크 상태 라우팅과 거리 벡터 라우팅의 차이를 설명하고, 각각 대표적으로 어떤 알고리즘을 사용하나요?

링크 상태 라우팅은 각 라우터가 네트워크 전체 토폴로지 정보를 알고 최단 경로를 계산하는 방식입니다. 대표적으로 다익스트라 알고리즘을 사용하며, OSPF가 이 방식에 해당합니다.

거리 벡터 라우팅은 각 라우터가 이웃 라우터와 거리 정보를 교환하면서 목적지까지의 비용을 갱신하는 방식입니다. 대표적으로 Bellman-Ford 알고리즘을 사용하며, RIP가 이 방식에 해당합니다.

## 13. IPv4 주소 체계에서 서브넷과 CIDR은 무엇이며, 왜 필요한가요?

서브넷은 하나의 IP 네트워크를 더 작은 네트워크 단위로 나눈 것입니다. 이를 통해 네트워크를 관리하기 쉽게 만들고, 브로드캐스트 범위를 제한할 수 있습니다.

CIDR은 `a.b.c.d/x` 형식으로 네트워크 prefix 길이를 유연하게 표현하는 방식입니다. 기존 클래스 기반 주소 체계보다 주소 공간을 효율적으로 사용할 수 있고, 라우팅 테이블을 줄이는 데도 도움이 됩니다.

## 14. ARP는 어떤 문제를 해결하기 위한 프로토콜이며, 같은 서브넷과 다른 서브넷으로 패킷을 보낼 때 MAC 주소는 어떻게 결정되나요?

ARP는 IP 주소에 대응되는 MAC 주소를 알아내기 위한 프로토콜입니다. 같은 서브넷의 호스트에게 보낼 때는 목적지 IP에 대한 ARP 요청을 브로드캐스트하고, 해당 호스트의 MAC 주소를 얻어 프레임을 전송합니다.

다른 서브넷으로 보낼 때는 목적지 호스트의 MAC 주소를 직접 찾는 것이 아니라 기본 게이트웨이, 즉 라우터 인터페이스의 MAC 주소를 ARP로 알아냅니다. 이후 라우터가 IP 패킷을 다음 네트워크로 전달합니다.

## 15. 대칭키 암호화, 공개키 암호화, 해시, MAC, 전자서명은 각각 어떤 보안 목표를 달성하기 위해 사용되나요?

대칭키 암호화는 송신자와 수신자가 같은 키를 사용해 데이터를 암호화하고 복호화하며, 주로 기밀성을 제공합니다. 공개키 암호화는 공개키와 개인키를 사용하며, 키 교환이나 인증에 유용하지만 대칭키보다 느립니다.

해시는 입력 데이터를 고정 길이의 값으로 바꾸어 무결성 검증에 사용됩니다. MAC은 해시와 비밀키를 함께 사용해 메시지 무결성과 송신자 인증을 제공합니다. 전자서명은 개인키로 서명하고 공개키로 검증하여 무결성, 송신자 인증, 부인 방지를 제공합니다.
