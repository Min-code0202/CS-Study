# 1. HTTP와 HTTPS의 차이는 무엇인가요?

- 답변
    
    HTTP는 서버와 클라이언트가 데이터를 주고받기 위한 프로토콜이며 데이터를 평문으로 전송합니다.
    
    HTTPS는 HTTP에 TLS(SSL)를 적용하여 데이터를 암호화한 프로토콜입니다.
    
    HTTPS는 데이터 암호화뿐 아니라 서버 인증과 데이터 무결성도 제공합니다.
    

---

### 꼬리질문 - HTTPS는 어떻게 동작하나요?

- 답변
    1. 클라이언트가 서버에 접속
    2. 서버가 인증서 전달
    3. 클라이언트가 인증서 검증
    4. 세션키 생성
    5. 세션키를 서버 공개키로 암호화하여 전달
    6. 서버가 개인키로 세션키 복호화
    7. 이후 대칭키 기반 통신

---

### 꼬리질문 - TLS란 무엇인가요?

- 답변
    
    TLS(Transport Layer Security)는 TCP 위에서 동작하는 보안 프로토콜입니다.
    
    기밀성, 무결성, 인증 기능을 제공합니다.
    
    HTTPS는 HTTP + TLS 조합입니다.
    

---

### 꼬리질문 - 인증서는 왜 필요한가요?

- 답변
    
    공개키가 진짜 서버의 것인지 증명하기 위해 필요합니다.
    
    인증서가 없다면 공격자가 자신의 공개키를 보내고 정상 서버인 척할 수 있습니다.
    

---

### 꼬리질문 - CA는 무엇인가요?

- 답변
    
    CA(Certificate Authority)는 인증 기관입니다.
    
    서버의 신원을 검증하고 인증서를 발급합니다.
    
    브라우저는 CA를 신뢰하기 때문에 인증서를 검증할 수 있습니다
    

---

### 꼬리질문 - HTTPS를 사용하면 완전히 안전한가요?

- 답변
    
    아닙니다.
    
    전송 구간은 보호할 수 있지만:
    
    - XSS
    - CSRF
    - SQL Injection
    - 인증 정보 탈취
    
    같은 공격은 별도로 방어해야 합니다.
    

---

# 2. TCP와 UDP의 차이는 무엇인가요?

- 답변
    
    TCP는 연결 지향 프로토콜이며 신뢰성을 보장합니다.
    
    UDP는 비연결형 프로토콜이며 신뢰성을 보장하지 않습니다.
    
    대신 UDP가 더 빠릅니다.
    
    | 구분 | TCP | UDP |
    | --- | --- | --- |
    | 연결 | 연결형 | 비연결형 |
    | 신뢰성 | 보장 | 보장 안함 |
    | 순서 보장 | O | X |
    | 속도 | 느림 | 빠름 |

---

### 꼬리질문 - TCP는 어떻게 신뢰성을 보장하나요?

- 답변
    - Sequence Number
    - ACK
    - 재전송
    - 흐름 제어
    - 혼잡 제어
    
    를 통해 신뢰성을 보장합니다.
    

---

### 꼬리질문 - UDP는 언제 사용하나요?

- 답변
    
    실시간성이 중요한 경우 사용합니다.
    
    예:
    
    - 실시간 스트리밍
    - 음성 통화
    - 온라인 게임
    - DNS

---

### 꼬리질문 - QUIC는 왜 UDP를 사용하나요?

- 답변
    
    TCP는 HOL Blocking 문제가 있습니다.
    
    QUIC는 UDP 위에서 신뢰성을 직접 구현하여 연결 수립 시간을 줄이고 HOL Blocking을 해결합니다.
    

---

# 3. TCP 3-Way Handshake를 설명해주세요.

- 답변
    
    TCP 연결을 생성하기 위한 과정입니다.
    
    1. Client → SYN
    2. Server → SYN + ACK
    3. Client → ACK
    
    이후 연결이 성립됩니다.
    

---

### 꼬리질문 - 왜 2-Way는 안되나요?

- 답변
    
    클라이언트가 서버의 수신 가능 여부는 확인할 수 있지만,
    
    서버가 클라이언트의 수신 가능 여부를 확인할 수 없습니다.
    
    따라서 양방향 통신 가능 여부를 확인하려면 3단계가 필요합니다.
    

---

### 꼬리질문 - SYN은 무엇인가요?

- 답변
    
    TCP 연결을 시작하겠다는 플래그입니다.
    
    초기 Sequence Number를 함께 전달합니다.
    

---

### 꼬리질문 - ACK는 무엇인가요?

- 답변
    
    상대방 패킷을 정상적으로 수신했다는 응답입니다.
    

---

### 꼬리질문 - SYN Flood 공격은 무엇인가요?

- 답변
    
    대량의 SYN 요청을 보내 서버의 연결 큐를 가득 채우는 공격입니다.
    
    정상 사용자가 접속하지 못하게 만듭니다.
    

---

# 4. TCP 4-Way Handshake를 설명해주세요.

- 답변
    
    TCP 연결을 종료하기 위한 과정입니다.
    
    1. Client → FIN
    2. Server → ACK
    3. Server → FIN
    4. Client → ACK
    
    이후 연결이 종료됩니다.
    

---

### 꼬리질문 - 왜 4번 통신하나요?

- 답변
    
    TCP는 양방향 연결입니다.
    
    각 방향을 독립적으로 종료해야 하기 때문에 4단계가 필요합니다.
    

---

### 꼬리질문 - TIME_WAIT는 무엇인가요?

- 답변
    
    연결 종료 후 일정 시간 대기하는 상태입니다.
    

---

### 꼬리질문 - TIME_WAIT가 필요한 이유는?

- 답변
    
    마지막 ACK가 유실되었을 경우 재전송된 FIN에 응답하기 위해 존재합니다.
    
    또한 네트워크에 남아있는 지연 패킷 제거 목적도 있습니다.
    

---

### 꼬리질문 - CLOSE_WAIT는 무엇인가요?

- 답변
    
    상대방으로부터 FIN을 받았지만 애플리케이션이 아직 소켓을 닫지 않은 상태입니다.
    
    CLOSE_WAIT가 많다면 서버 코드 문제일 가능성이 높습니다.
    

---

# 5. HTTP 1.1 / 2.0 / 3.0 차이를 설명해주세요.

- 답변
    
    HTTP는 성능 개선을 위해 지속적으로 발전했습니다.
    
    ---
    
    ### HTTP 1.1
    
    특징
    
    - Keep Alive 지원
    - Pipelining 지원
    
    문제점
    
    - HOL Blocking
    - Header 중복 전송
    
    ---
    
    ### HTTP 2.0
    
    특징
    
    - 멀티플렉싱
    - Header Compression
    - Server Push
    
    장점
    
    - 하나의 연결에서 여러 요청 처리 가능
    
    ---
    
    ### HTTP 3.0
    
    특징
    
    - QUIC 사용
    - UDP 기반
    
    장점
    
    - TCP HOL Blocking 해결
    - 빠른 연결 수립

---

### 꼬리질문 - HOL Blocking이란?

- 답변
    
    앞 요청이 지연되면 뒤 요청도 함께 대기하는 현상입니다.
    

---

### 꼬리질문 - 멀티플렉싱이란?

- 답변
    
    하나의 연결에서 여러 요청과 응답을 동시에 처리하는 기술입니다.
    

---

### 꼬리질문 - HTTP 2.0도 HOL Blocking이 발생하나요?

- 답변
    
    애플리케이션 레벨에서는 해결했지만 TCP 레벨에서는 여전히 발생할 수 있습니다.
    

---

### 꼬리질문 - QUIC란 무엇인가요?

- 답변
    
    Google이 개발한 UDP 기반 전송 프로토콜입니다.
    
    HTTP 3.0의 기반이 되며:
    
    - 빠른 연결 수립
    - HOL Blocking 해결
    - TLS 내장
    
    특징을 가집니다.
    

---

## 6. 브라우저에 URL을 입력하면 어떤 일이 발생하나요?

- 답변
    1. 브라우저가 DNS 조회를 수행한다.
    2. 도메인에 대응되는 IP 주소를 얻는다.
    3. 같은 서브넷이 아니라면 게이트웨이 MAC 주소를 ARP로 조회한다.
    4. TCP 3-Way Handshake를 수행한다.
    5. HTTPS라면 TLS Handshake를 수행한다.
    6. HTTP Request를 서버로 전송한다.
    7. 서버가 HTTP Response를 반환한다.
    8. 브라우저가 HTML, CSS, JS를 파싱한다.
    9. 렌더 트리를 생성하고 화면에 출력한다.

### 꼬리질문 - DNS 조회 과정은?

- 답변
    
    브라우저 캐시 확인
    
    → OS 캐시 확인
    
    → 로컬 DNS 서버 확인
    
    → Root DNS
    
    → TLD DNS
    
    → Authoritative DNS
    
    순으로 조회한다.
    

### 꼬리질문 - ARP는 언제 사용되나요?

- 답변
    
    같은 LAN에서 목적지 MAC 주소를 알아야 할 때 사용한다.
    

### 꼬리질문 - TCP 연결은 언제 생성되나요?

- 답변
    
    DNS 조회가 끝나고 서버 IP를 알게 된 이후 생성된다.
    

### 꼬리질문 - HTTPS라면 TLS는 언제 수행되나요?

- 답변
    
    TCP 연결이 완료된 이후 수행된다.
    
    즉
    
    DNS
    
    → TCP
    
    → TLS
    
    → HTTP
    
    순서다.
    

### 꼬리질문 - 브라우저 렌더링은 어떻게 이루어지나요?

- 답변
    
    HTML 파싱
    
    → DOM 생성
    
    → CSSOM 생성
    
    → Render Tree 생성
    
    → Layout
    
    → Paint
    
    순서로 진행된다.
    

---

## 7. DNS는 무엇인가요?

- 답변
    
    DNS(Domain Name System)는 사람이 이해하기 쉬운 도메인 이름을 IP 주소로 변환하는 시스템이다.
    
    예:
    
    www.naver.com
    
    ↓
    
    223.xxx.xxx.xxx
    

### 꼬리질문 - DNS 조회 과정 설명해주세요.

- 답변
    
    브라우저 캐시
    
    → OS 캐시
    
    → 로컬 DNS 서버
    
    → Root DNS
    
    → TLD DNS
    
    → Authoritative DNS
    
    순으로 조회한다.
    

### 꼬리질문 - Recursive Query와 Iterative Query 차이는?

- 답변
    
    Recursive Query
    
    DNS 서버가 최종 결과를 찾아서 반환
    
    Iterative Query
    
    다음 DNS 서버 정보를 반환
    

### 꼬리질문 - DNS 캐싱은 어디서 이루어지나요?

- 답변
    - 브라우저
    - 운영체제
    - 로컬 DNS 서버
    
    에서 수행된다.
    

### 꼬리질문 - DNS Spoofing은 무엇인가요?

- 답변
    
    공격자가 가짜 DNS 응답을 보내 사용자를 악성 사이트로 유도하는 공격이다.

## 8. 쿠키와 세션의 차이는 무엇인가요?

- 답변
    
    쿠키는 클라이언트에 저장되는 데이터이다.
    
    세션은 서버에 저장되는 사용자 상태 정보이다.
    
    일반적으로 세션 ID를 쿠키에 저장한다.
    

### 꼬리질문 - HTTP는 왜 Stateless인가요?

- 답변
    
    서버가 클라이언트 상태를 저장하지 않기 때문이다.
    
    요청 하나하나를 독립적으로 처리한다.
    

### 꼬리질문 - 세션은 어디에 저장되나요?

- 답변
    - 메모리
    - DB
    - Redis
    
    등에 저장할 수 있다.
    

### 꼬리질문 - 세션 방식의 단점은?

- 답변
    
    사용자가 많아질수록 서버 메모리 사용량이 증가한다.
    

### 꼬리질문 - 세션 클러스터링 문제는?

- 답변
    
    서버가 여러 대인 경우 사용자의 세션이 다른 서버에 존재할 수 있다.
    

### 꼬리질문 - Redis를 사용하는 이유는?

- 답변
    
    세션을 중앙 저장소에 보관하여 여러 서버가 공유할 수 있게 하기 위함이다.
    

---

## 9. JWT란 무엇인가요?

- 답변
    
    JWT(Json Web Token)는 사용자 정보를 담고 있는 토큰 기반 인증 방식이다.
    
    서버가 상태를 저장하지 않아도 된다.
    
    ### 구조
    
    Header.Payload.Signature
    

### 꼬리질문 - JWT의 장점은?

- 답변
    - Stateless
    - 확장성 우수
    - 서버 부하 감소

### 꼬리질문 - JWT의 단점은?

- 답변
    - 토큰 탈취 시 위험
    - 강제 로그아웃 어려움
    - Payload 노출 가능

### 꼬리질문 - Refresh Token은 왜 필요한가요?

- 답변
    
    Access Token 탈취 위험을 줄이기 위해 짧은 수명을 사용하고 재발급용으로 Refresh Token을 사용한다.
    

### 꼬리질문 - 세션과 JWT의 차이는?

- 답변
    
    세션
    
    - Stateful
    - 서버 저장
    
    JWT
    
    - Stateless
    - 클라이언트 저장

## 10. 대칭키 암호화와 공개키 암호화의 차이는 무엇인가요?

- 답변
    
    대칭키는 암호화와 복호화에 같은 키를 사용한다.
    
    공개키는 공개키와 개인키를 사용한다.
    

### 꼬리질문 - 대칭키의 장단점은?

- 답변
    
    장점
    
    - 빠르다
    
    단점
    
    - 키 공유 문제
    
    대표
    
    - AES

### 꼬리질문 - 공개키의 장단점은?

- 답변
    
    장점
    
    - 안전한 키 교환 가능
    
    단점
    
    - 느리다
    
    대표
    
    - RSA

### 꼬리질문 - 왜 공개키만 사용하지 않나요?

- 답변
    
    공개키 암호화는 연산 비용이 매우 크기 때문이다.
    

### 꼬리질문 - 왜 대칭키만 사용하지 않나요?

- 답변
    
    안전하게 키를 전달할 방법이 없기 때문이다.
    

### 꼬리질문 - TLS는 왜 둘을 함께 사용하나요?

- 답변
    
    공개키로 세션키를 교환하고
    
    대칭키로 실제 데이터를 암호화한다.
    

### 꼬리질문 - 세션키는 무엇인가요?

- 답변
    
    통신 중에만 사용하는 임시 대칭키이다.
