3.1 트랜스포트 계층

각기 **다른 호스트**에서 동작하는 애플리케이션 프로세스 간의 논리적 통신(logical commmunication)을 제공

트랜스포트 계층 프로토콜은 네트워크 라우터가 아니라 end-system에서 구현됨.

sender : application의 message를 조각들로 쪼개고 각 조각들에 헤더를 붙여서 (=segments 단위로 쪼갠 후) 네트워크 계층으로 전달
receiver : segments들 조립한 후 application layer로 전달.

3.1.1 transport & network layer
네트워크 : 호스트들 간의 논리적 통신(communication)
    * 호스트(컴퓨터) A → 호스트 B까지 데이터 전달
    * "어느 길로 갈지(라우팅)", "어떻게 전달할지" 담당

트랜스포트 : 프로세스들 간의 통신(communication) - 다음 서비스에 대해 신경 안씀 (path, 방법 등은 신경 안쓰고 오로지 보내는거에만 신경!) 다음 계층 내부에서 어떻게 이동하는지 언급 x
    * 호스트 안의 프로세스 A → 프로세스 B까지 전달
    * 네트워크 계층이 어떻게 동작하는지 신경 안 씀 그냥 "받아서 올바른 프로세스에 넘겨주면 끝"

3.1.2 TCP & UDP
TCP (Transmission Control Protocol)
* 데이터를 신뢰성 있게, 순차적 (in-order delivery)으로 전달
* 혼잡제어 : 네트워크 혼잡을 방지하기 위해 데이터 전송속도를 조절함 
* 흐름제어 (flow control) : receiver와 sender 간의 데이터 전송 속도를 조절하여 receiver가 과부하되지 않도록함
* connection oriented(연결지향형) : 데이터를 전송하기 전에 연결을 미리 설정함

UDP (User Datagram Protocol) 
* 데이터를 신뢰성 없이, 순서에 상관없이(unordered delivery) 전달
* connection-less(비연결형) : 사전에 연결 설정하지 않음
* 참고) 신뢰성 없는데 쓰는 이유 -> 빠르다! (게임, 스트리밍 같은 서비스에 사용됨)

TCP,UPD가 제공하지 않는 것
1. 전송 지연 (delay)을 보장하지 않음 
    -> "이 데이터 100ms 안에 도착시켜줘" 같은 요청을 들어줄 수 없다.
2. 대역폭(단위 시간당 보낼 수 있는 데이터 양)을 보장하지 않음.
    -> "이 앱한테는 항상 10Mbps 보장해줘" 같은 요청을 못 들어줌.

3.2 다중화(multiplexing)와 역다중화(demultiplexing)

서버에는 multiple clients랑 connection이 있는데 어느 소켓으로 보낼지 어떻게 아는가? => Port번호 필요함.IP는 기계를 나타내고 port번호는 프로세스 나타냄.

역다중화(demultiplexing): 수신된 세그먼트를 올바른 소켓으로 전달
다중화(multiplexing): 소켓에서 데이터를 모아 세그먼트 생성 후 네트워크 계층으로 전송

1. dest 호스트(receiver)에서 트랜스포트 게층은 세그먼트를 수신함. 
2. 트랜스포트 계층은 세그먼트를 올바른 소켓으로 전달함.
    * 프로세스는 소켓(데이터 출입구)을 가지고있음
    * 소켓은 식별자로 포트 번호를 가짐.
3. 트랜스포트 계층의 세그먼트는 특별한 필드를 갖는다.
    * 출발지 포트 번호
    * 목적지 포트 번호

multiplexing at sender(N -> 1) : multiple 소켓들로부터 데이터를 관리하고 트랜스포트 헤더를 붙임.(demux에 쓰임)
demultiplexing at receiver : 전달받은 세그먼트를 올바른 소켓으로 전달하기 위해 헤더정도를 사용함.

![alt text](image.png)

demux 작동법 (1 -> N)
- 세그먼트는 source, destination 포트 번호를 갖고있음. 데이터그램에는 source, destination IP주소를 갖고있음.
- 호스트의 각 소켓은 포트 번호를 할당 받음.
- 세그먼트가 호스트에 도착하면, 목적지 포트번호를 검사하고 세그먼트를 보냄.
- 즉 호스트는 IP 주소(기계)랑 포트번호(프로세스)를 사용해서 적절한 소켓에 세그먼트를 전달함.

비연결형 (connectionless) demultiplexing (UDP)
UDP 소켓은 목적지 IP주소와 포트번호로 식별된다.
-> 다른 source IP와 포트 번호를 가진 UDP 데이터그램(세그먼트)가 같은 목적지 포트 번호, 같은 목적지 IP주소를 갖고있으면 같은 소켓으로 direct됨 (즉 1:1 맵핑이 아님.)
-> 결국 같은 dest로 가는거면 왜 출발지(source) 포트 번호는 왜 필요한가? : 회신할 수도 있으니까.

연결형 (connection-oriented) (TCP)
- 데이터를 전송하기 전에 connection이 필요함.
- TCP 소켓은 4가지로 소켓을 구분함. (이 중 하나라도 다르면 다른 소켓과 연결됨)
  - 출발지(SOURCE) IP주소
  - 출발지 포트번호
  - 도착지(DEST) IP주소
  - 도착지 포트번호

TCP 연결 설정 
1. TCP서버 애플리케이션은 welcome 소켓(예 : 포트번호 12000)을 갖고있음. TCP 클라이언트로부터 연결 설정 요청을 기다림.
2. TCP 클라이언트는 소켓을 생성하고 연결 설정요청 세그먼트를 보냄.
3. 서버 프로세스 동작중인 컴퓨터의 호스트가 목적지 포트12000을 포함하는 연결 요청 세그먼트를수신
4. 포트번호 12000으로 연결 수락을 기다리는 서버 프로세스로 보냄
5. 서버는 연결 요청 세그먼트의 4개 요소를 주목함
    - 도착하는 세그먼트의 출발지 포트, 출발지 IP 주소, 목적지 포트,목적지 IP 주소가 전부 일치하면,그 세그먼트는 이 소켓으로 역다중화될 것이다.


3.3 UDP - User Datagram Protocol
순서
1. 메시지로부터 출발지포트와 목적지 포트번호를 첨부함.
2. 출발지 호스트의 IP, 목적지 호스트의 IP를 추가한 후 세그먼트를 네트워크로 넘겨줌
3. 세그먼트를 IP 데이터그램으로 캡슐화 후 receiver 호스트에게 전달
4. 호스트에게 도착하면 UDP는 어플리케이션 프로세스(소켓)에 전달하기 위해 목적지 포트 번호 사용함.(demultiplexes msg)
-> 트랜스포트 계층이 할 수 있는 최소 기능으로 동작함
-> UDP segments는 유실될 수 있음 (= not reliable)
-> 순차적으로 app에 deliver되지 않음(out of order)
-> connectionless
    - udp sender와 reciever사이에 handshaking없음.
-> 왜 써?
    - connection 설립 필요없음. (RTT 딜레이x)
    - 단순함
    - 어플리케이션 레벨에서 정교한 데이터 제어 가능 (TCP에는 혼잡 제어 메커니즘이 있어서 호스트들 사이 링크가 혼잡해지면 TCP 송신자를 제한함. 또 시간에 관계없이 수신여부를 응답받을 때 까지 세그먼트를 계속 재전송하여 시간이 오래 걸림.)
    - 헤더 사이즈가 작음 (8 byte, tcp는 세그먼트마다 20byte의 헤더) 
    - but 혼잡 제어가 없음 
      - 필요한 작업을 할 수 없게 되는 상황 발생 가능(폭주 상태로 인해)
-> 만약 신뢰적인 전송이 필요하면..
    1. application 계층에서 신뢰성을 추가
    2. application 계층에서 혼잡 제어를 추가해준다.
DNS : 전형적으로 UDP 사용하는 어플리케이션 프로토콜. 만약 호스트가 응답 수신 못하면 다른 네임서버로 송신하거나, 어플리케이션에게 응답수신없음을 통보.
SNMP, HTTP/3, 실시간 스트리밍

세그먼트가 출발지에서부터 목적지로 이동할 때 세그먼트 안에 내용이 변경된게 있는지 검사할 때 사용됨.

  * 그냥 버리거나 경고랑 같이 넘겨서 app이 처리하도록 함.

end-end principle : 복잡한 기능은 네트워크 중간이 아니라 양쪽 끝에서 처리하자! 
-> 데이터 전송 중 오류가 발생했다면 중간(라우터)에서 처리하지 말고 끝단에서 처리하자! 따라서 udp는 아무것도 안하고 그냥 전달만 함. 

3.4 신뢰적인 데이터 전송의 원리
= no packet loss, no packet error, in order delivery
- 실제 application layer밑에 link들은 신뢰적이지 않음. 그래서 트랜스포트 프로토콜은 이런 환경 고려해야한다.
- 또 receiver가 msg 잘 받았는지 확인 해야하는 경우도 있다. 이 역시 transport가 고려해야함.

RDT (Reliable Data Transfer)
각 버전 별로 읽어보기👍
rdt1.0 - 완전히 신뢰적인 채널
rdt2.0 - 비트 오류가 있는 채널 ((ACK/NAK), NAK받으면 재전송)
rdt2.1 - 순서 번호 추가
rdt2.2 - NAK 없는 버전 (같은 ACK가 두 번 오면(중복 ACK) sender가 NAK와 동일하게 해석. )
rdt3.0 - 패킷 손실까지 처리
비트 오류뿐만 아니라 패킷 자체가 손실되는 경우도 처리. 타이머(timer) 추가. 일정 시간 내에 ACK가 안 오면 재전송. 타임아웃 발생 이유는 두 가지. 패킷이 지연된 경우(조기 타임아웃), ACK가 손실된 경우.
==> stop-and-wait 방식이라 한 번에 하나씩만 전송 -> 채널 이용률이 매우 낮음.

파이프라이닝(pipelining) : ACK를 기다리지 않고 여러 패킷을 연속으로 전송. 채널 이용률 크게 향상.

GBN (Go-Back-N, N부터 다시 가기)
윈도 크기 N개까지 ACK 없이 연속 전송 가능.
receiver는 순서대로 받다가 순서가 잘못된 패킷은 버리고 마지막으로 정상 수신한 패킷에 대한 ACK 재전송.
오류 발생 시 해당 패킷부터 윈도우 내 모든 패킷 재전송 (불필요한 재전송이 많아지는게 단점)
수신자쪽 구현이 단순함. (버퍼링 필요 없음.)

SR (Selective Repeat, 선택적 반복)
오류가 난 패킷만 재전송. GBN처럼 불필요한 재전송 없음.
receiver가 순서 바뀐 패킷도 버퍼에 저장해두고, 빠진 패킷이 도착하면 순서대로 상위 계층에 전달.
각 패킷마다 개별 타이머와 개별 ACK 필요. 수신자 구현이 GBN보다 복잡함.

3.5 TCP
- 일대일 (1 sender, 1 receiver) : 멀티 캐스팅 불가
- 전이중 (full-duplex) 양쪽이 동시에 데이터를 주고받을 수 있음
- 연결 지향형 : 데이터 전송 전에 반드시 handshaking해야함!
- flow가 control된다. : sender는 receiver를 overwhelm하지 않는다
- pipelining : TCP 혼잡이랑 흐름제어는 window의 사이즈를 고려함. (receiver의 버퍼 크기 고려)

순서 번호와 확인응답 번호
![numbers](img/numbers.jpeg)
TCP는 데이터를 구조화 되어있지 않고 단지 순서대로 정렬되어있는 byte스트림으로 본다. 
TCP는 **전이중 방식**임을 상기하자.(호스트 A가 호스트 B로 데이터를 송신하는 **동안에** 호스트 B로부터 데이터를 수신하게 해줌)
순서 번호 (Sequence numbers) : 세그먼트 첫 번째 바이트의 바이트 스트림 번호. TCP는 데이터를 바이트 스트림으로 봄. (내가 보내는 데이터에 번호 붙임)
확인응답 번호 (Acknowledgements) : 수신자가 다음으로 받기를 기대하는 바이트 번호. ex) 0~535를 받았으면 확인응답 번호는 536. (상대방이 보낸 데이터에 대해 ACK)
```
A → B : [순서번호: 100, ACK번호: 536]
순서번호 100 → "내(A)가 보내는 데이터는 100번 바이트부터 시작"
ACK번호 536 → "나(A)는 너(B)의 데이터 535번까지 받았어, 536번 줘"
B → A : [순서번호: 536, ACK번호: 200]
순서번호 536 → "내(B)가 보내는 데이터는 536번 바이트부터 시작"
ACK번호 200 → "나(B)는 너(A)의 데이터 199번까지 받았어, 200번 줘"

-> 피기배킹 : 데이터를 보내면서 ACK도 같이 싫어보냄.
```
![telnet](img/telnet.png)

누적 확인응답(cumulative ACK) : 첫 번째 잃어버린 바이트 이전까지만 확인응답. 순서 바뀐 세그먼트는 버퍼에 저장하고 빈 부분이 채워지면 상위 계층에 전달.

만약 세그먼트의 순서가 다르게 도착한다면? -> 규칙같은건 정해두지 않음. TCP구현 개발자에게 맡김
1. 순서가 바뀐 세그먼트 버리기
2. 순서 바뀐 세그먼트 보관하고있다가 빈 공간에 잃어버린 데이터 채우기위해 기다림. (버퍼 저장 한 후 기다리기) <- 이 방법 주로 사용함.

3.5.3 왕복 시간(RTT) 예측과 타임아웃
RTT(Round-Trip Time) : 세그먼트 전송부터 ACK 받을 때까지 걸리는 시간.
타임아웃은 RTT를 기반으로 동적으로 결정됨. 고정값이 아니라 네트워크 상황이 나빠지면 타임아웃도 같이 늘어남. 타임아웃이 너무 짧으면 불필요한 재전송 발생, 너무 길면 손실 감지가 늦어짐.
RTT가 길다는 건 그만큼 응답이 느리다는 뜻. 그래서 서버를 사용자 가까이 두거나 CDN(Content Delivery Network)을 쓰는 게 RTT를 줄이는 방법.

3.5.4 신뢰적인 데이터 전송

TCP 재전송
타임아웃 재전송 : 타이머 만료 시 해당 세그먼트 재전송. ACK가 손실되더라도 누적 ACK덕분에 불필요한 재전송 줄일 ㅅ ㅜ있음 
```
예시 
A 입장 (송신자)
세그먼트1(0~4)  전송 →
세그먼트2(5~9)  전송 →
A는 그냥 순서대로 보내기만 함.

B 입장 (수신자)
세그먼트1(0~4)  도착 → ACK 5  전송 (손실)
세그먼트2(5~9)  도착 → ACK 10 전송 (도착)
B는 둘 다 받았기 때문에 ACK 10을 보낼 수 있음.

A 입장 (다시)
ACK 5  → 못 받음
ACK 10 → 받음
A는 ACK 5를 못 받았지만, ACK 10이 왔으니까 B가 0~9를 다 받았음을 알 수 있음.
```

빠른 재전송(fast retransmit) : 타임아웃 기다리지 않고 중복 ACK 3개 수신 시 즉시 재전송. 중복 ACK는 수신자가 기대한 것보다 높은 순서 번호(seq #)의 세그먼트가 왔을 때 발생.
```
예시) 
세그먼트1(0~4)   도착 → ACK 5
세그먼트2(5~9)   손실
세그먼트3(10~14) 도착 → ACK 5 (중복 1)
세그먼트4(15~19) 도착 → ACK 5 (중복 2)
세그먼트5(20~24) 도착 → ACK 5 (중복 3)
```

TCP는 GBN + SR 혼합 방식 사용. 기본적으로 누적 ACK(GBN 방식)지만 선택적 확인응답(SACK-어디가 빠졌는지 정확히 알려줌, 손실된 것만 재전송) 옵션으로 SR처럼 동작 가능.

3.5.5 흐름 제어
-> 보내는 속도가 app이 소켓을 읽는 속도보다 빠르다면? recv버퍼에 data쌓임, 그러다 버퍼 벗어나면 유실됨 -> 난리남 -> 흐름 조절 해야함!
수신자(receiver)버퍼 오버플로 방지하기 위해 수신자가 현재 버의 여유공간(receive window)를 세그먼트 헤더에 담아 sender에게 알려줌.
TCP는 전이중(full-duplex)이므로 연결의 각 측의 sender는 별개의 rwnd를 가짐
sender는 확인응답 안된 데이터 양을 rwnd 이하로 유지.
호스트 A가 호스트 B에게 큰 파일을 전송한다고 가정
* 호스트 B는 이 연결에 수신 버퍼(RcvBuffer)를 할당함
* 호스트 B의 application 프로세스는 버퍼로부터 데이터를 읽으면서 변수 정의함.
(LastByteRead : 호스트 B의 애플리케이션 프로세스에 의해 버퍼로부터 읽힌 데이터 스트림의 마지막 바이트 번호)
(LastByteRcvd : 호스트 B에서 네트워크로부터 도착하여 수신 버퍼에 저장된 데이터 스트림의 마지막 바이트 번호)
LastByteRcvd - LastByteRead ≤ RcvBuffer
rwnd = RcvBuffer - (LastByteRcvd - LastByteRead)
rwnd = 0이면 송신자는 1바이트짜리 세그먼트를 계속 보내서 버퍼가 비워지면 알 수 있게 함.


흐름 제어 vs 혼잡 제어 : 혼잡 제어는 네트워크 자체가 감당할 수 있는 수준으로 송신 속도를 조절하는 것. 흐름 제어는 수신자 버퍼가 넘치지 않게 조절하는 것.
흐름 제어는 수신자(receiver) 버퍼 보호, 혼잡 제어는 네트워크 혼잡 방지. 목적이 다름.

3.5.6 TCP 연결 관리

3-way Handshake
![3way](img/threeway.png)

1단계 : 클라이언트 -> 서버. SYN세그먼트 전송 (SYN=1), 최소 순서 번호(client_isn)를 임의로 선택하고, 순서 번호 필드에 넣어서 전송.
2단계 : 서버 -> 클라이언트. 연결 승인 SYNACK 세그먼트 전송. SYN = 1, ACK = client_isn+1, 서버 초기 순서번호 (server_isn)포함. 서버가 버퍼와 변수 할당함. 
3단계 : 클라이언트 → 서버. ACK 전송. SYN=0, ACK=server_isn+1. 클라이언트가 버퍼와 변수 할당. 이 세그먼트부터 데이터 포함 가능.
이 과정에서 양쪽이 서로 초기 순서 번호를 교환하고, 버퍼를 할당해서 통신 준비를 마침.

참고) 왜 2-way가 아니라 3-way?
-> 양방향 통신을 하기 위해서!
2-way면 클라이언트만 "서버가 내 말을 들을 수 있다"는 걸 확인할 수 있음. 근데 서버 입장에서는 "클라이언트가 내 말을 들을 수 있는지"를 확인할 방법이 없음.
3단계에서 클라이언트가 ACK를 보내야 서버도 "아 클라이언트가 내 응답을 받았구나, 양방향 통신이 가능하다"는 걸 확인할 수 있음.


연결의 종료 
![closing](img/closing.png)

1. 클라이언트 프로세스 종료명령을 내림 (FIN 비트를 1로 설정)
2. 서버는 확인 응답 세그먼트를 보냄
3. 서버는 close 후 FIN 비트가 1로 설정된 자신의 종료 세그먼트를 보냄.
4. 클라이언트는 서버에 ACK 전송 후 연결 종료. 자원 회수.

SYN 플러드 공격 : 3-way handshake의 3단계(ACK)를 보내지 않고 SYN만 대량으로 보냄. 서버가 half-open 연결에 자원을 계속 할당하다가 소진 → 정상 클라이언트 서비스 거부(DoS).

3.6 혼잡 제어의 원리 - 라우터의 상황을 고려해서 보내자1 
혼잡 : 너무 많은 송신자가 너무 빠른 속도로 데이터를 보내서 네트워크가 감당을 못하는 상태. 패킷 손실과 큐잉 지연이 늘어나는 게 증상.

시나리오 1. 버퍼가 무한함.
![congestion1](img/congestion1.jpeg)

- 패킷이 버려지지는 않지만 링크 용량에 가까워질수록 큐잉 지연이 급격히 커짐. 아무리 빠르게 보내도 각 연결이 얻을 수 있는 처리량은 링크 용량의 절반이 한계.

시나리오 2. 버퍼가 유한함.
![congestion2](img/congestion2.jpeg)

-  버퍼가 꽉 차면 패킷 손실 발생 → 재전송 필요 → 재전송 트래픽이 쌓여서 실제 유효 처리량은 더 줄어듦. 불필요한 재전송(아직 안 버려진 패킷을 조기 타임아웃으로 재전송)이 발생하면 링크 대역폭이 낭비됨.
![congestion2](img/congestion_sc2.png)
-> 재전송으로 인해 실제 유효 처리량이 줄어듦. 

시나리오 3. 멀티홉 경로
![congestion3](img/congestion3.jpeg)

- 패킷이 여러 라우터를 거치다가 마지막에 버려지면 그 패킷을 전달하느라 앞쪽 라우터들이 쓴 대역폭이 전부 낭비됨. 혼잡이 심해지면 종단 간 처리율이 0에 수렴할 수도 있음.
![congestion3](img/sc3.png)


혼잡 제어 접근법 2가지

1. 종단 간 혼잡 제어(end-end congestion control)
종단 시스템(end system)이 패킷 손실이나 지연 증가를 보고 스스로 혼잡을 추론해서 전송 속도를 줄임. TCP가 사용하는 방식.
참고) TCP가 혼잡을 감지하는 방법
패킷 손실(타임아웃 또는 중복 ACK 3개)이 발생하거나 RTT가 증가하는 걸 보고 혼잡이 생겼다고 판단. 라우터가 직접 알려주지 않아도 종단에서 스스로 추측하는 방식.

2. 네트워크 지원 혼잡 제어 (Network-assisted congestion control)
라우터가 혼잡 상태를 직접 송신자나 수신자에게 알려줌. 라우터가 송신자에게 직접 초크 패킷(choke packet)을 보내거나, 패킷 헤더에 혼잡 표시를 해서 수신자가 이를 송신자에게 전달하는 방식.
