트랜스포트 계층 서비스 및 개요

애플리케이션 프로세스 간의 논리적 통신을 제공 -> 동작하는 호스트들이 직접 연결된 것처럼 보임

트랜스포트 계층 프로토콜은 네트워크 라우터가 아닌 호스트에서 구현됨

1. 송신 측의 트랜스포트 수신 받은 메시지를 트랜스포트 계층 패킷(segment)로 변환
2. 트랜스포트 계층은 네트워크 계층으로 세그먼트 전달, 저번에 네트워크 5계층에서 본 것처럼 헤더를 씌워서 전달됨
3. 수신 측 네트워크 계층은 세그먼트 추출해서 트랜스포트 계층으로 세그먼트 보내고, 트랜스포트 계층에서 애플리케이션 계층에서 이용할 수 있도록 세그먼트 처리

네트워크 애플리케이션에서는 하나 이상의 트랜스포트 계층 프로토콜을 사용할 수 있음
예: TCP, UDP

트랜스포트 계층과 네트워크 계층 사이의 관계
* 트랜스포트 계층 -> 다른 호스트에서 동작하는 프로세스들 사이의 논리적 통신을 제공

* 네트워크 계층 프로토콜 -> 호스트들 사이의 논리적 통신을 제공

트랜스포트 계층은 네트워크 계층이 못 하는 것 일부는 보완할 수 있다.
예: IP 프로토콜은 비신뢰적이지만, TCP는 기능을 추가해서 신뢰성을 만들어낼 수 있다.
하지만 네트워크 자원 자체에 의존하는 지연, 대역폭 같은 보장은 혼자 만들 수 없다.

다중화와 역다중화
네트워크 계층이 동작하는 호스트 대 호스트 전달 서비스에서
호스트에서 동작하는 애플리케이션에 대한 프로세스 대 프로세스 전달 서비스로 확장한는 과정

1. 목적지에서 트랜스포트 계층은 네트워크 계층으로부터 세그먼트 수신
2. 트랜스포트 계층은 세그먼트를 소켓에게 전달
    프로세스는 애플리케이션의 한 부분으로서 소켓을 가지고 있음 -> 소켓은 데이터를 전달하는 출입구 역할을 가지며 유일한 식별자, 포트 번호를 가짐

역다중화란? 세그먼트를 올바른 소켓으로 전달하는 작업 -> UDP의 동작 과정과 같다
다중화란? 출발지 호스트의 소켓으로부터 데이터를 모으고, 세그먼트 생성을 위해 각 데이터에 헤더 정보로 캡슐화, 세그먼트들을 네트워크 계층으로 전달

다중화 요구사항
1. 소켓은 유일한 포트 번호를 가짐
2. 각 세그먼트는 세그먼트가 전달될 적절한 소켓을 가리키는 특별한 필드를 가짐
    출발지 포트 번호 필드
    도착지 포트 번호 필드

출발지 포트 번호는 어디에 사용되는가?
    회신 주소의 한 부분으로 사용됨

UDP의 장점
1. 무슨 데이터를 언제 보낼지에 대해서 애플리케이션 레벨에서 더 정교한 제어가 가능하다. -> 실시간 애플리케이션에서는 UDP를 사용하고, 필요한 추가 기능을 구현할 수 있다.
UDP는 비신뢰적이지만 애플리케이션 자체에 신뢰성을 제공하면 신뢰적인 데이터 전송이 가능하다.

2. 연결 설정이 없다 -> 연결을 위한 어떤 지연도 없다.
연결 설정이 없으므로 연결 상태 또한 없다. 자원을 덜 사용하기 때문에 더 많은 액티브 클라이언트를 수용할 수 있다고 이해
해킷 헤더에 오버헤드가 적다

UDP의 단점
혼잡 제어를 하지 않는다. 
이 때,
1. 라우터에 많은 패킷 오버 플로우가 발생해서 소수의 패킷만이 출발지-목적지 간의 경로를 통과할 수 있다. -> 패킷 손실
2. 이 때 발생한 높은 손실률은 그 손실률을 감소시키기 위해 TCP 송신자들이 속도를 줄이도록 할 것이며 이는 TCP 세션의 혼잡을 유래한다.

2. 포트 번호
3. 체크섬
4. 길이 필드

체크섬 -> 오류 검사를 위한 것
왜 체크섬을 제공하는 가?
출발지와 목적지 사이의 모든 링크가 오류 검사를 제공한다는 보장이 없기 때문이다 -> 모든 경로에서 오류 검사를 하지 않을 수도 있다로 이해
세그먼트들이 정확하게 링크를 통해 전송되었을지라도, 세그먼트가 라우터의 메모리에 저장될 때 비트 오류가 발생할 수 있다.

다만 UDP는 오류 검사를 제공하지만, 오류 회복을 위한 어떠한 일도 하지 않음
예: 손상된 세그먼트를 그냥 버리거나, 경고와 함께 손상된 세그먼트를 넘겨주기도

* 신뢰적인 채널에서는 전송된 데이터가 손상되거나 손실되지 않으며,

연결지향형 트랜스포트: TCP

TCP 연결
TCP는 애플리케이션 프로세스가 데이터를 다른 프로세스에게 보내기 전에, 두 프로세스가 서로 '핸드셰이크'를 먼저 해야 하므로 연결지향형(connection-oriented)이다.

TCP 연결은 전이중 서비스(full-duplex service)를 제공한다.
만약 호스트 A의 프로세스와 호스트 B의 프로세스 사이에 TCP 연결이 있다면, 애플리케이션 계층 데이터는 B에서 A로 흐르는 동시에 A에서 B로 흐를 수 있다.

TCP 연결은 점대점이다 -> 따라서 단일 송신 동작으로 한 송신자가 여러 수신자에게 데이터를 전송하는 멀티캐스팅은 TCP에서 불가능

TCP 연결 과정
* client process -> 연결을 먼저 요청하는 쪽
* server process -> 연결 요청을 기다리고 있다가 받아주는 쪽

연결을 three-way handshake로 이루어진다.
연결 과정
1. 클라이언트 애플리케이션 프로세스는 서버의 프로세스와 연결을 설정하기를 원한다고 TCP 클라이언트에게 먼저 알림
2. 클라이언트의 트랜스포트 계층은 서버 TCP와의 TCP 연결 설정을 진행 -> 클라이언트가 특별한 TCP 세그먼트를 보냄
3. 서버는 두번째 특별한 TCP 세그먼트로 응답
4. 클라이언트가 세번째 특별한 세그먼트로 다시 응답

처음 2개의 세그먼트에는 데이터가 없고, 세 번째에는 페이로드를 포함할 수 있다.

연결이 설정되면 2개의 애플리케이션 프로세스는 서로 데이터를 보낼 수 있다.
1. 클라이언트 프로세스는 소켓을 통해 데이터 스트림을 전달
2. 데이터가 소켓을 통해 전달되면, 이제 데이터는 TCP에 맡겨짐
-> 버퍼 중 하나인 연결의 송신 버퍼로 데이터 보냄
-> 때때로 TCP는 송신 버퍼에서 데이터 묶음을 만들어서 보낸다

최대 세그먼트 크기(maximum segment size, MSS)
세그먼트로 모아 담을 수 있는 최대 데이터 양은 MSS로 제한됨
-> 로컬 송신 호스트에 의해 전송될 수 있는 가장 큰 링크 계층 프레임의 길이(최대 전송 단위-maximum transmission unit, MTU)

-> 헤더를 포함하는 TCP 세그먼트의 최대 크기가 x
-> 세그먼트에 있는 애플리케이션 계층 데이터에 대한 최대 크기

TCP 세그먼트
TCP 헤더 + 클라이언트 데이터
세그먼트는 네트워크로 송신된다 -> TCP 세그먼트가 IP 데이터그램 안에 담겨서 외부 네트워크로 전송된다는 의미

1. 순서 번호
TCP는 데이터를 구조화되어 있지 않고 단지 순서대로 정렬되어 있는 바이트 스트림으로 본다.
세그먼트에 대한 순서 번호는 세그먼트에 있는 첫 번째 바이트의 바이트 스트림 번호

예: 데이터 스트림은 500,000 바이트로 구성된 파일이라고 가정한다.
MSS는 1,000 바이트
데이터 스트림의 첫 번째 바이트는 0으로 설정하고, 그 이후의 각각의 세그먼트 내부의 순서 번호 필드에 삽입

# 순서 번호 이미지

2. 확인 응답 번호
TCP는 전이중 방식 -> A가 B로 데이터 송신하는 동안에 B로부터 데이터를 수신 받을 수 있음

B로부터 도착한 각 세그먼트는 B로부터 A로 들어온 데이터에 대한 순서 번호를 가짐
-> A가 자신의 세그먼트에 삽입하는 확인 응답 번호는 A가 B로부터 기대하는 다음 바이트의 순서 번호임 -> ACK만 담은 세그먼트를 보낼 수 잇음

B에서 A로 데이터가 이동
순서 번호: 100
데이터 크기 20byte
-> A는 B로부터 바이트 번호 100 ~ 119까지 받은 것임
-> A가 B에게 ACK 번호가 120이라고 알려줘야 함 -> 119번까지 잘 받았다, 120번 바이트부터 보내면 된다 -> 다음에 받고 싶은 바이트 번호

TCP는 스트림에서 첫 번째 잃어버린 바이트까지의 바이트들까지만 확인 응답 -> 누적 확인 응답을 제공

만약에 위의 예시에서 100~119까지 받고, 120에서 139 수신되기 전에 140~159을 수신하면 순서가 틀려지게 됨

-> 이 때 호스트가 해야할 행동에 대해서 어떠한 것도 정해져 있지 않음 -> TCP 구현 개발자가 알아서 하도록

1. 수신자가 세그먼트 버림
2. 수신자가 순서 바뀐 데이터 보유, 빈 공간에 잃어버린 데이터 채우기 위해 기다림 -> 실제로 이렇게 함

예: 탈넷
# 탈넷 이미지

1. 42번 보낸다, 73번 잘 받았다, 데이터는 'C'임
2. 79번 보낸다, 42잘 받았고 43 다음에 줘라, 데이터 'C' 받았었음 -> 이 때 피기백이란? 업혀가다는 뜻으로, ACK만 따로 보내지 않고,
내가 상대에게 보내는 데이터 세그먼트에 ACK를 같이 실어 보내는 것
3. 43번 보낸다, 79 잘 받았고, 80번 다음에 줘라

RTT(왕복 시간)과 타임아웃
TCP는 손실 세그먼트 발견을 위해 타임아웃/재전송 매커니즘 사용
-> 타임아웃은 연결의 RTT 보다 좀 더 커야 한다.

왕복 시간을 예측할 때는 전송되었지만 현재까지 확인 응답이 없는 세그먼트 중 하나에 대해서만 측정
-> 1, 2, 3, 4를 보냈다고 할 때 아직 응답이 안온 1에 대해서 측정
-> rtt를 계산할 때는 평균을 사용
-> 왜냐하면 라우터의 혼잡과 종단 시스템에서의 부하 변화 때문에 세그먼트마다 값이 달라질 수 있기 때문에

재전송 타임아웃 주기의 설정과 관리
주기는 EstimatedRTT 보다 크거나 같아야 함
-> 그렇지 않으면 불필요한 재전송이 보내짐
-> EstimatedRTT는 위에서 구한 RTT의 가중평균(weighted average)이다.

너무 크면 안됨
-> 너무 크면 세그먼트를 잃었을 때, 즉각적인 재전송이 안됨

=> 타임아웃값은 Estimated에 약간의 여윳값을 더한 값으로 설정

신뢰적인 데이터 전송
신뢰적인 데이터 전송이란?
-> 프로세스가 지신의 수신 버퍼로부터 읽은 데이터 스트림이 손상되지 않음을 보장
-> 중복이 없다는 것과 순서가 유지된다는 것을 보장
=> 바이트 스트림은 송신자가 전송한 것과 같은 바이트 스트림

타이머 관리
-> 다수의 세그먼트가 아니라 하나의 세그먼트에 대해서 재전송 타이머 사용(단일 타이머)

# 이미지

흐름 제어
TCP는 송신자가 수신자의 버퍼를 오버플로시키는 것을 방지하기 위해서 흐름 제어 서비스를 제공
-> 수신하는 애플리케이션이 읽는 속도와 전송하는 속도를 같게 한다.

수신 윈도우(receive window)라는 변수를 통해 흐름 제어 제공
-> 수신 측에서 가용한 버퍼 공간이 얼마나 되는지를 송신자에게 알려줌
-> 전이중이므로 각 측의 송신자는 별개의 수신 위도를 유지

# 수신 윈도우 이미지

호스트 B는 호스트 B가 호스트 A에게 전송하는 모든 세그먼트의 윈도 필드에 현재 rwnd 값을 설정한다.

이를 통해 호스트 A에게 연결 버퍼에 얼마만큼의 여유 공간이 있는지를 알려준다.

호스트 A는 두 변수 LastByteSent와 LastByteAcked를 유지한다.

LastByteSent - LastByteAcked = 호스트 A가 이 연결에 전송 확인응답이 안 된 데이터의 양

버퍼 여유 공간보다 작은 확인 응답 안 된 데이터 양을 유지해서 오버플로 발생하지 않도록

문제점과 해결법
B의 수신 버퍼가 0이고 B가 A에게 전송할 것이 없는 경우
-> B가 버퍼를 비워도 버퍼 여유 공간이 생겼다라고 전달할 수 없음
-> A가 B에세 데이터 보낼 수가 없다
-> A가 B의 수신 윈도우가 0일 때, 1 바이트 데이터로 세그먼트를 계속해서 전송하도록 함

혼잡 제어의 원리
혼잡의 원인: 너무 많은 출발지가 너무 높은 속도로 데이터를 보내려고 시도할 때 발생

혼잡 제어에 대한 접근법
네트워크 계층은 혼잡 제어 목적을 위해 트랜스포트 계층에게 어떤 지원도 제공 X

TCP가 혼잡 제어를 위해 취하는 방식
* TCP 세그먼트 손실과 증가하는 왕복 지연값을 혼잡의 발생 표시로 간주 -> 그에 따라서 윈도 크기를 줄인다.
-> 혼잡 윈도(cwnd)를 추적해서 TCP 송신자가 네트워크로 전송할 수 있는 속도에 제약을 가한다.

cwnd값을 조정하는 매커니즘


네트워크 지원 혼잡 제어
1. 직접 피드백
    네트워크 라우터가 송신자에세 초크 패킷을 보내서 알림
2. 송신자에서 수신자로 흐르는 패킷안에 특정 필드에 표시/수정
    수신자가 표시된 패킷을 수신하고 혼잡 상태를 송신자에게 알림
