## 신뢰적인 데이터 전송의 원리

### 신뢰적인 데이터 전송
> 비신뢰적인 네트워크 위에서 데이터가 손실, 손상, 중복, 순서 뒤바뀜 없이 전달되도록 만드는 방식

- 트랜스포트 계층뿐 아니라 링크 계층, 애플리케이션 계층에서도 필요한 개념
- TCP가 제공하는 핵심 서비스가 신뢰적인 데이터 전송이다

---

### 신뢰적인 채널이 보장하는 것

- 데이터 손상 없음
- 데이터 손실 없음
- 중복 데이터 없음
- 전송한 순서 그대로 전달

즉, 송신자가 보낸 데이터 스트림과 수신자가 읽는 데이터 스트림이 같아야 한다.

---

### 왜 필요한가

IP는 best-effort 서비스이기 때문에 다음을 보장하지 않는다.

- 전달 보장 없음
- 순서 보장 없음
- 데이터 무결성 보장 없음

따라서 TCP 같은 트랜스포트 계층 프로토콜이 신뢰성을 직접 구현해야 한다.

---

## 신뢰성을 위한 핵심 메커니즘

### Checksum
> 전송 중 데이터에 비트 오류가 발생했는지 검사하는 값

- 수신자는 checksum을 이용해 데이터 손상 여부를 확인한다
- 오류를 발견하는 역할이지, 오류를 직접 복구하는 것은 아니다

---

### ACK
> 수신자가 데이터를 정상적으로 받았음을 송신자에게 알려주는 확인응답

- 송신자는 ACK를 통해 어떤 데이터가 정상적으로 도착했는지 알 수 있다
- TCP에서는 누적 ACK 방식을 사용한다

---

### Retransmission
> 데이터 또는 ACK가 손실되었다고 판단되면 다시 전송하는 방식

- 송신자는 일정 시간 동안 ACK가 오지 않으면 재전송한다
- 이를 통해 패킷 손실을 복구한다

---

### Sequence Number
> 데이터의 순서를 구분하기 위한 번호

- 수신자는 sequence number로 데이터 순서를 맞춘다
- 중복 패킷인지 새로운 패킷인지 구분할 수 있다
- TCP에서는 바이트 단위로 sequence number를 관리한다

---

## Stop-and-Wait

### Stop-and-Wait 방식
> 패킷 하나를 보내고 ACK가 올 때까지 기다리는 방식

- 구현이 단순하다
- 한 번에 하나의 패킷만 전송한다
- ACK를 받기 전까지 다음 패킷을 보낼 수 없다

---

### 문제점
- 대기 시간이 길다
- 네트워크 대역폭을 충분히 활용하지 못한다
- RTT가 큰 환경에서는 성능이 크게 떨어진다

즉, 신뢰성은 보장할 수 있지만 전송 효율이 낮다.

---

## Timeout

### Timeout
> 일정 시간 동안 ACK가 오지 않으면 패킷이 손실되었다고 판단하는 방식

- 송신자는 패킷을 보낸 뒤 타이머를 시작한다
- ACK가 도착하면 타이머를 종료한다
- 타이머가 만료되면 해당 패킷을 재전송한다

---

### Timeout이 필요한 이유
송신자는 다음 상황을 직접 구분할 수 없다.

- 데이터 패킷이 손실된 경우
- ACK가 손실된 경우
- 네트워크 지연으로 ACK가 늦게 도착하는 경우

따라서 일정 시간이 지나면 손실로 간주하고 재전송한다.

---

### Timeout 설정 시 주의점

- 너무 짧으면 -> 실제로는 손실되지 않았는데 불필요하게 재전송
- 너무 길면 -> 손실 복구가 늦어져 전체 지연 증가

---

## Pipelining

### Pipelining
> ACK를 기다리지 않고 여러 패킷을 연속해서 전송하는 방식

- Stop-and-Wait의 비효율을 개선한다
- 여러 패킷을 동시에 전송 중인 상태로 둘 수 있다
- 네트워크 이용률이 증가한다

---

### 필요한 이유
Stop-and-Wait는 하나의 패킷을 보내고 ACK를 기다리기 때문에 대역폭을 제대로 활용하지 못한다.

Pipelining은 여러 패킷을 미리 보내서 네트워크 경로를 계속 채워두므로 전송 효율을 높일 수 있다.

---

### Pipelining을 위해 필요한 것

- 더 넓은 sequence number 범위
- 송신 측 버퍼
- 수신 측 버퍼
- window 기반 전송 제어

여러 패킷이 동시에 전송 중이면 각각을 구분해야 하므로 sequence number가 필요하고, ACK를 받기 전까지 데이터를 보관해야 하므로 버퍼가 필요하다.

---

## Sliding Window

### Sliding Window
> ACK를 받지 않은 패킷 중 전송 가능한 범위를 window로 관리하는 방식

- window 안에 있는 패킷은 전송 가능
- ACK를 받으면 window가 앞으로 이동한다
- 한 번에 여러 패킷을 전송할 수 있게 해준다

---

### Window가 필요한 이유
무제한으로 패킷을 보내면 수신자 버퍼나 네트워크가 감당하지 못할 수 있다.

따라서 송신자는 window 크기만큼만 미확인 데이터를 전송한다.

---

## 오류 복구 방식

### Go-Back-N
> 손실된 패킷 이후의 패킷들을 모두 다시 전송하는 방식

- 송신자는 여러 패킷을 연속해서 보낼 수 있다
- 수신자는 순서가 맞는 패킷만 받아들인다
- 중간에 하나가 손실되면 그 뒤의 패킷들은 버려질 수 있다
- 손실된 패킷부터 다시 전송한다

---

### Go-Back-N 특징
- 구현이 단순하다
- 수신자 버퍼 관리가 쉽다
- 하지만 불필요한 재전송이 많이 발생할 수 있다

예를 들어 2번 패킷만 손실되었는데 3, 4, 5번 패킷까지 다시 보내야 할 수 있다.

---

### Selective Repeat
> 손실되거나 오류가 발생한 패킷만 선택적으로 다시 전송하는 방식

- 수신자는 순서가 맞지 않는 패킷도 버퍼에 저장할 수 있다
- 손실된 패킷이 나중에 도착하면 저장해둔 패킷들과 함께 순서대로 상위 계층에 전달한다
- 필요한 패킷만 재전송하므로 효율적이다

---

### Selective Repeat 특징
- 불필요한 재전송을 줄일 수 있다
- 네트워크 대역폭을 더 효율적으로 사용한다
- 대신 수신자 버퍼 관리와 ACK 관리가 복잡하다

---

## Go-Back-N vs Selective Repeat

| 구분 | Go-Back-N | Selective Repeat |
|------|-----------|------------------|
| 재전송 방식 | 손실된 패킷 이후 모두 재전송 | 손실된 패킷만 재전송 |
| 수신자 처리 | 순서가 맞지 않으면 버림 | 순서가 달라도 버퍼에 저장 |
| ACK 방식 | 누적 ACK 중심 | 개별 ACK 중심 |
| 장점 | 구현 단순 | 전송 효율 좋음 |
| 단점 | 불필요한 재전송 많음 | 구현 복잡 |

---

## 핵심 정리

### 신뢰적인 데이터 전송을 구성하는 요소

- Checksum -> 오류 검출
- ACK -> 정상 수신 확인
- Retransmission -> 손실 복구
- Sequence Number -> 순서 보장, 중복 제거
- Timeout -> ACK 미수신 시 재전송 판단
- Window / Pipelining -> 전송 효율 개선

---

### 최종 정리
- IP는 신뢰성을 보장하지 않는다
- TCP는 비신뢰적인 IP 위에서 신뢰적인 전송을 구현한다
- 이를 위해 ACK, 재전송, sequence number, checksum을 사용한다
- Stop-and-Wait는 단순하지만 비효율적이다
- Pipelining과 Sliding Window는 전송 효율을 높이기 위한 방식이다
- Go-Back-N과 Selective Repeat은 손실이 발생했을 때의 복구 방식 차이다
