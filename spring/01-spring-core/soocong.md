# DI / IoC

## 1. IoC란 무엇인가요?

A. IoC는 제어의 역전이라는 개념으로, 객체 생성과 의존성 관리의 책임을 개발자에게서 Spring 컨테이너로 넘기는 방식입니다. 즉, 객체가 스스로 의존 관계를 만들기보다, Spring이 적절한 객체를 연결해 주도록 만드는 구조라고 이해하면 됩니다.

## 2. DI란 무엇인가요?

A. DI는 의존성 주입으로, 객체가 직접 의존 객체를 생성하거나 관리하지 않고 외부에서 주입받도록 하는 방식입니다. 이 방식을 사용하면 객체 간 결합도를 낮추고, 변경과 테스트가 쉬운 구조를 만들 수 있습니다.

### 꼬리질문

#### 1) 생성자 주입을 권장하는 이유는 무엇인가요?

A. 생성자 주입은 의존 관계가 클래스 생성 시점에 명확하게 드러나기 때문에 코드가 더 읽기 쉽고, 객체를 불변으로 만들기에도 유리합니다. 또한 테스트 코드에서 Mock 객체를 주입하기도 편하고, 의존성이 빠뜨려지지 않도록 해줘서 실무에서 가장 안정적인 방식으로 많이 권장됩니다.

#### 2) @Primary와 @Qualifier의 차이는 무엇인가요?

A. @Primary는 여러 구현체가 있을 때 기본적으로 우선 사용할 빈을 지정하는 역할을 합니다. 반면 @Qualifier는 특정 빈 이름이나 식별자를 명시해 원하는 구현체를 정확하게 선택하는 방식입니다. 즉, 기본값을 정하느냐, 특정 대상을 직접 고르느냐의 차이라고 보면 됩니다.

#### 3) 구현체가 여러 개라면 Spring은 어떤 Bean을 주입하나요?

A. 여러 구현체가 존재하면 Spring은 기본적으로 어떤 빈을 주입할지 결정하지 못해 ambiguity 문제를 일으킵니다. 따라서 @Primary를 사용해 기본 선택지를 두거나, @Qualifier로 특정 빈을 명시해 주어야 합니다.

#### 4) 객체를 직접 new로 생성하는 것과 Bean을 주입받는 것의 차이는 무엇인가요?

A. new로 생성하면 객체를 개발자가 직접 관리해야 하고, 의존 관계도 코드에서 직접 연결해 주어야 합니다. 반면 Spring Bean으로 주입받으면 Spring이 객체의 생명주기와 의존성을 관리해 주기 때문에 코드가 더 유연하고 확장성 있게 설계됩니다.

#### 5) AOP란?

A. AOP는 로깅, 트랜잭션, 권한 검사처럼 여러 비즈니스 로직에 공통으로 사용되는 기능을 분리해 관리하는 프로그래밍 방식입니다. Spring에서는 주로 프록시를 통해 메서드 호출을 가로채고, 실제 로직 실행 전후에 공통 기능을 적용합니다.

# Bean

## 3. 일반 객체와 Spring Bean의 차이는 무엇인가요?

A. 일반 객체는 단순히 클래스의 인스턴스이며, 생성과 소멸을 개발자가 직접 관리합니다. 반면 Spring Bean은 Spring 컨테이너가 관리하는 객체로, 생성 시점, 의존성 주입, 생명주기 콜백까지 컨테이너가 책임집니다.

## 4. Bean을 등록하는 방법에는 어떤 것들이 있나요?

A. 가장 대표적인 방법은 @ComponentScan을 통해 클래스 패스상에서 자동으로 빈을 등록하는 방식입니다. 또 다른 방법으로는 @Configuration 클래스 안에서 @Bean 메서드를 선언해 직접 빈을 등록하는 수동 등록 방식이 있습니다.

## 5. @Component와 @Bean의 차이는 무엇인가요?

A. @Component는 클래스 수준에 붙여 Spring이 자동으로 스캔해서 빈으로 등록하게 하는 방식입니다. 반면 @Bean은 메서드 수준에서 직접 객체를 반환하도록 하며, 외부 라이브러리나 복잡한 초기화가 필요한 객체를 등록할 때 주로 사용됩니다.

### 꼬리질문

#### 1) Bean의 기본 이름은 어떻게 결정되나요?

A. 빈의 기본 이름은 보통 클래스 이름을 기준으로 결정됩니다. 별도로 @Bean(name = "...")처럼 이름을 지정할 수도 있습니다.

#### 2) Bean으로 등록되지 않은 객체에는 의존성 주입이 가능한가요?

A. 아니요. 의존성 주입은 Spring 컨테이너가 관리하는 빈 객체에 대해서만 적용됩니다. 일반적으로 new로 생성한 객체는 컨테이너가 관리하지 않으므로, 해당 객체에는 주입이 되지 않습니다.

# Bean Lifecycle

## 6. Spring Bean의 생명주기를 설명해주세요.

A. Spring Bean은 보통 생성 단계, 초기화 단계, 사용 단계, 소멸 단계의 순서로 관리됩니다. 빈이 생성된 뒤 초기화 메서드가 호출되고, 애플리케이션 종료 시점에 소멸 메서드가 실행됩니다. 이를 통해 객체가 안전하게 준비되고 정리될 수 있습니다.

### 꼬리질문

#### 1) 프록시 객체는 Bean 생명주기 중 언제 만들어지나요?

A. 프록시 객체는 보통 빈이 생성된 뒤, AOP가 적용되는 시점에 동적으로 만들어집니다. 즉, 실제 빈이 생성된 다음에 프록시가 감싸서 동작하게 되며, 이 시점은 AOP 설정과 빈 생성 방식에 따라 달라질 수 있습니다.

# Bean Scope

## 7. Spring Bean의 기본 Scope는 무엇인가요?

A. Spring Bean의 기본 Scope는 singleton입니다. 즉, 같은 빈에 대해 애플리케이션 컨테이너 안에서 하나의 인스턴스만 생성하고 공유해서 사용하게 됩니다.

## 8. Singleton과 Prototype Scope의 차이는 무엇인가요?

A. Singleton Scope는 빈을 한 번만 생성해 재사용하는 방식이고, Prototype Scope는 빈을 요청할 때마다 새로 생성하는 방식입니다. 따라서 상태를 자주 바꾸는 객체나 매 요청마다 독립적인 객체가 필요한 경우에는 Prototype이 더 적합합니다.

### 꼬리질문

#### 1) Singleton Bean에서 상태를 가지면 안 되는 이유는 무엇인가요?

A. Singleton 빈은 여러 요청이나 스레드가 같은 인스턴스를 공유하기 때문에, 필드에 상태를 저장하면 데이터가 섞이거나 오염될 수 있습니다. 그래서 싱글톤 빈은 주로 stateless하게 설계하는 것이 좋습니다.

#### 2) Spring Singleton과 디자인 패턴의 Singleton은 어떤 차이가 있나요?

A. 디자인 패턴의 Singleton은 보통 특정 클래스가 하나의 인스턴스만 가지도록 강제하는 개념입니다. 반면 Spring의 Singleton은 컨테이너가 관리하는 빈의 범위에 해당하며, 애플리케이션 컨텍스트 안에서 하나의 인스턴스로 관리된다는 점이 다릅니다.

# Component Scan

## 9. Component Scan이란 무엇인가요?

A. Component Scan은 지정한 패키지를 기준으로 @Component, @Service, @Repository, @Controller 같은 애너테이션이 붙은 클래스를 자동으로 찾아 빈으로 등록하는 과정입니다. 이를 통해 개발자가 일일이 빈을 등록하지 않아도 되도록 도와줍니다.

## 10. Spring은 어떤 클래스를 자동으로 Bean으로 등록하나요?

A. 보통 @Component가 붙은 클래스와 그를 포함하는 @Service, @Repository, @Controller 같은 스테레오타입 애너테이션이 붙은 클래스를 자동으로 빈으로 등록합니다. 이 애너테이션들은 모두 컴포넌트 스캔의 대상이 됩니다.

## 11. Bean 이름이 중복되면 어떻게 되나요?

A. 빈 이름이 중복되면 Spring은 충돌을 감지해 예외를 발생시킵니다. 이런 상황은 빈 이름을 명시적으로 지정하거나, 중복 등록된 빈 중 하나를 제거해서 해결해야 합니다.

### 꼬리질문

#### 1) 다른 패키지의 Component를 스캔하려면 어떻게 해야 하나요?

A. @ComponentScan 애너테이션에 basePackages 또는 basePackageClasses 속성을 지정하면 됩니다. 이렇게 하면 기본 스캔 범위를 넓히거나 특정 패키지만 대상으로 좁혀서 컴포넌트를 찾을 수 있습니다.

# Spring Boot

## 12. Spring과 Spring Boot의 차이는 무엇인가요?

A. Spring은 핵심 프레임워크 자체를 의미하며, 설정을 직접 구성해야 하는 경우가 많습니다. 반면 Spring Boot는 이러한 설정을 자동화하고 기본적인 의존성을 쉽게 구성해 주기 때문에, 개발자가 비즈니스 로직에 더 집중할 수 있게 해줍니다.

## 13. 자동 설정, Auto Configuration이란 무엇인가요?

A. Auto Configuration은 클래스패스에 있는 라이브러리와 설정 조건을 기반으로, 필요한 빈들을 자동으로 만들어 주는 기능입니다. 예를 들어 데이터소스나 웹 서버 관련 설정을 개발자가 거의 직접 하지 않아도 되도록 도와줍니다.

## 14. @SpringBootApplication을 설명해주세요.

A. @SpringBootApplication은 Spring Boot 애플리케이션의 진입점이 되는 애너테이션입니다. 내부적으로 @Configuration, @EnableAutoConfiguration, @ComponentScan을 함께 포함해, 애플리케이션 구동에 필요한 기본 설정을 한 번에 처리해 줍니다.

### 꼬리질문

#### 1) JAR와 WAR 배포의 차이는 무엇인가요?

A. JAR는 실행 가능한 애플리케이션 형태로 패키징되어 독립적으로 실행하기 쉽고, WAR는 웹 애플리케이션 서버에 배포되는 형태입니다. 일반적으로 Spring Boot는 JAR 형태로 배포하는 경우가 많습니다.

#### 2) Profile은 무엇이고 어떻게 분리하나요?

A. Profile은 개발, 테스트, 운영 등 환경별로 설정을 분리하기 위한 기능입니다. 예를 들어 application-dev.yml, application-prod.yml처럼 파일을 나누고, 실행 환경에 따라 활성화된 프로필을 선택해 사용할 수 있습니다.

#### 3) Spring Boot가 편리한 대신 가지는 단점은 무엇인가요?

A. Spring Boot는 자동 설정이 편리하지만, 그만큼 내부 동작을 추상화해 놓기 때문에 원리를 깊게 이해하지 못하면 문제가 생겼을 때 원인을 파악하기 어려울 수 있습니다. 또한 커스터마이징이 필요해지면 설정이 복잡해질 수도 있습니다.
